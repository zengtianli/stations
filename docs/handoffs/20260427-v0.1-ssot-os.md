## 踩过的坑

### 1. auggie binary 版本兼容
- brew 0.24.0 跟 augmentproxy 反代协议**不兼容**（全 timeout）
- nvm 0.19.0 在反代上工作
- **2026-04-27 切官方 Augment Indie Plan** 后，两个 binary 应都行；脚本仍锁 nvm 兜底

### 2. CLI flag 组合冲突
- `-p -a -q --output-format json -i ... -w` → 卡死（-q 和 --output-format json 并存）
- 删 `-q`（quiet 多余 — JSON 输出本身单行 result）→ 修复
- 教训：远端 CLI 工具调用先二分缩小责任 flag，不要拷复合命令直接用

### 3. 多 MCP server 残留
- 检查发现 4 个 `auggie --mcp` 进程同时存在（CC 历史 session 残留）
- kill 后无差别（augmentproxy 反代当时也挂了）
- 警示：每 CC session spawn 1 MCP server，多 session 可能堆叠（v0.x 加 mcp-gc skill）

### 4. 反代频率限流自伤
- 25 路并发 MCP retrieval 试错 + 多次 CLI 试错 → augmentproxy 反代触发 throttle
- 用户先于我感知（"这 auggie 怎么很慢"）
- 教训：付费远端 API 不要乱并发；augment cold-start 串行+耐心

### 5. 大仓索引超时根因
- 即使 .gitignore 排除 target/.next/node_modules（augment doc 确认 honor），cold-start 大仓仍 > 90s
- 真原因：augment 后端要跑 embedding，5-12G 真代码（即使排除 bloat）embedding 计算就需要时间
- 教训：A1 acceptance "全部 ≥ 0.7 relevance" 不切实际，要分仓大小分级

### 6. 并行优先 vs 远端 API 节流的冲突
- 用户全局 CLAUDE.md 写"并行优先"是针对**本地工具**（Read/Grep/Bash）
- 远端**收费 API**（auggie / Claude API）反而要节流（rate limit 敏感）
- 我没区分清楚是 v0.1 几个失败的根因之一

### 7. /warmup 错过 ROADMAP.md
- `/warmup` 脚本只读 HANDOFF，不读 ROADMAP — 多轮项目情况下用户得手动提醒
- 候选 v0.2 改进：`/warmup` 检测同目录有 ROADMAP.md 时自动列摘要
