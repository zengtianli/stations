Command 模块化重构计划

# Command 模块化重构计划 ## Context 18 个全局命令存在重叠和硬编码问题。目标：最大化模块化，每个命令做一件事，组合命令通过调用原子命令编排流程。未来可自由组合出新功能。 用户额外需求：想按 dashboard 页面维度来维护（repos/vps/tasks/app 各自可刷新），不是只有一个大 /deploy。 ## 最终命令架构 ``` 原子命令（一个命令做一件事，可独立用，也可被组合） ├── /ship commit + push ├── /promote GitHub metadata（description/topics/homepage/screenshot） ├── /tidy 目录整理（垃圾/版本冗余/散乱文件） ├── /scan 扫描 CC 会话 → tasks.json ★ 新 ├── /pull 批量 git pull --ff-only ★ 新 ├── /audit repo 完整性审计 ★ 新 ├── /vps-status VPS 巡检 ├── /cf-dns DNS 管理 └── /repo-map repo 注册表 组合命令（编排原子命令） ├── /dashboard 按页面维护 dashboard 数据 + 部署 ★ 重写 ├── /deploy 通用部署（任意有 deploy.sh 的项目） ★ 重写 ├── /groom repo 维护 = /pull → /audit → fix(/promote) → /ship ★ 重写 └── /health 健康检查 = 扫描 → [/tidy] ★ 重写 领域命令（不变） ├── /md2word, /review-docx ├── /build-hydro, /migrate-hydro └── /context, /recap 移动 └── /frontend command → skill（自动触发） ``` ### 组合关系图 ``` /dashboard ─┬─ repos → /repo-map scan + /pull repo-dashboard ├─ tasks → /scan ├─ vps → /vps-status ├─ app → /deploy（部署 dashboard 本身） └─ (无参) → 以上全部 + /deploy /deploy ────── deploy.sh → verify（通用，任意项目） /groom ─┬─ /pull (Phase 1) ├─ /audit (Phase 2) ├─ /promote (Phase 3: metadata fix) └─ /ship (Phase 5) /health ─── /tidy (发现问题时调用) 自由组合示例： /pull all + /audit --check = 快速巡检所有 repo /scan + /repo-map sync = 刷新 dashboard 全部数据源不部署 /audit + /promote = 只查 + 只修 metadata /health all --fix = 全 repo 卫生扫描 + 自动清理 ``` --- ## 具体变更 ### 1. 新建 `/scan` **文件**: `~/.claude/commands/scan.md`（新建） ``` 扫描 Claude Code 会话，LLM 分析任务状态，生成 tasks.json。 1. python3 ~/Dev/repo-dashboard/lib/task_analyzer.py 2. 输出：任务数量、状态分布、新增/变更任务 3. 结果写入 ~/Dev/configs/tasks.json $ARGUMENTS: --force 忽略缓存，重新分析所有会话 --dry-run 只分析不写入文件 ``` ### 2. 新建 `/pull` **文件**: `~/.claude/commands/pull.md`（新建） ``` 批量 git pull --ff-only。 范围（同 /ship 的参数规范）： 无参数 → 当前目录 <name> [name2] → repo-map.json 查路径 all → 所有 non-ignored repo 输出： ✓ dockit pulled (2 commits) ✓ hydro-rainfall already up to date ⚠ vps diverged — manual resolve needed — oauth-proxy no remote tracking branch ``` ### 3. 新建 `/audit` **文件**: `~/.claude/commands/audit.md`（新建） ``` 审计 repo 完整性（基于 repo-standards.json）。 范围：同 /pull --category <cat> 按分类过滤 检查项： - LICENSE / .gitignore / README.md / README_CN.md / CLAUDE.md 存在性 - README 结构符合 category 模板 - GitHub metadata（description/topics/homepage） 参考： - ~/Dev/configs/repo-standards.json - python3 ~/Dev/devtools/repo_manager.py audit 输出： ✅ dockit — all clear 🔧 hydro-rainfall — missing LICENSE, missing README_CN.md ⚠️ essays — missing LICENSE, no description ``` ### 4. 重写 `/dashboard`（按页面维护入口） **重写**: `~/.claude/commands/dashboard.md` ``` Dashboard 数据维护 + 部署。按页面刷新数据或全量更新。 $ARGUMENTS 指定页面： repos → 刷新 repo 数据 1. /repo-map scan（发现新 repo） 2. /pull repo-dashboard（拉最新代码） 3. 输出 repo-map.json 变更摘要 tasks → 刷新任务数据 1. /scan（扫描 CC 会话 → tasks.json） 2. 输出任务状态分布 vps → 检查 VPS 状态 1. /vps-status（服务/端口/磁盘/内存/Docker） 2. 输出异常项 app → 部署 dashboard 1. cd ~/Dev/repo-dashboard && /deploy 2. 验证 https://dashboard.tianlizeng.cloud (无参数) → 全量：repos + tasks + vps + app 按顺序执行以上全部，最后部署 每步完成后输出一行状态，最终汇总表。 ``` ### 5. 重写 `/deploy`（通用化，去硬编码） **重写**: `~/.claude/commands/deploy.md` ``` 通用部署，适配任意有 deploy.sh 的项目。 1. 确认当前目录有 deploy.sh 2. 前置步骤（自动检测）： - 有 frontend/ 且有未构建变更 → npm run build 3. bash deploy.sh 4. 验证（从 CLAUDE.md 自动提取）： - 线上地址 → curl -sI 检查 HTTP 200 - 服务名 → ssh VPS systemctl status $ARGUMENTS: --no-verify 跳过验证 --dry-run 只显示会执行什么，不实际执行 凭证在 ~/.personal_env，不问用户要。 ``` ### 6. 重写 `/health`（全局化 + 调用 /tidy） **重写**: `~/.claude/commands/health.md` ``` 文件卫生 + Git 状态检查。发现问题可调用 /tidy 清理。 范围： 无参数 → 当前目录 <name> → repo-map.json 查路径 all → 所有 repo 检查项： 1. Git 状态 - uncommitted changes - behind remote（git fetch --dry-run） - untracked 大文件（>10MB） - 敏感文件被 track（.env、*.pem、*.key） 2. 文件卫生 - 垃圾文件：~$*、.DS_Store、Thumbs.db、__pycache__ - 版本冗余（deliverables/docs/ 下的版本链） 3. ~/Work/ 目录额外检查 - zdwp 下调用 zdwp_health.py 输出后若有可清理项 → 问用户是否调用 /tidy。 $ARGUMENTS: --fix 自动调用 /tidy 清理（不二次确认） ``` ### 7. 重写 `/groom`（编排原子命令） **重写**: `~/.claude/commands/groom.md` 保留 5 阶段结构，但显式引用原子命令： ``` repo 维护全流程。 Phase 1: /pull <targets> Phase 2: /audit <targets> Phase 3: Fix - LICENSE/.gitignore → 自动生成 - GitHub metadata → /promote <name> - CLAUDE.md/README → 生成（需逐个确认） Phase 4: Review checkpoint（批量 + 逐个确认） Phase 5: /ship <targets> $ARGUMENTS: 无参数 → all <name> [name2] → 指定 repo --check → 只跑 /pull + /audit，不修复 --skip-pull → 跳过 Phase 1 --category <cat> → 按分类过滤 引用的原子命令：/pull, /audit, /promote, /ship 引用的数据源：repo-map.json, repo-standards.json ``` /groom 的详细 Phase 3-4 规则（LICENSE 模板、README 模板、Review 格式等）保持不变，只是开头和结构改为显式引用原子命令。 ### 8. `/frontend` → skill **删除**: `~/.claude/commands/frontend.md` **新建**: `~/Dev/cc-harness/skills/frontend.md` ```yaml --- description: 前端开发规范 trigger: 当构建 Streamlit / Web 前端 UI 时自动加载 --- ``` 内容不变（亮色主题、紧凑布局、可操作性、Streamlit 规范）。 ### 9. `/promote` 保持独立（不变） 独立可用 + 被 /groom Phase 3 调用。内容不改。 --- ## 变更文件清单 | 操作 | 文件 | |------|------| | 新建 | `~/.claude/commands/scan.md` | | 新建 | `~/.claude/commands/pull.md` | | 新建 | `~/.claude/commands/audit.md` | | 重写 | `~/.claude/commands/dashboard.md` | | 重写 | `~/.claude/commands/deploy.md` | | 重写 | `~/.claude/commands/health.md` | | 重写 | `~/.claude/commands/groom.md` | | 删除 | `~/.claude/commands/frontend.md` | | 新建 | `~/Dev/cc-harness/skills/frontend.md` | 不变的命令（9 个）：`/ship`, `/promote`, `/tidy`, `/vps-status`, `/cf-dns`, `/repo-map`, `/md2word`, `/review-docx`, `/build-hydro`, `/migrate-hydro`, `/context`, `/recap` ## 验证 1. `/scan` → tasks.json 生成 2. `/pull doctools` → 单 repo pull 3. `/audit doctools` → 单 repo 审计 4. `/dashboard repos` → repo-map scan + pull 5. `/dashboard tasks` → scan CC 会话 6. `/dashboard vps` → VPS 状态检查 7. `/dashboard app` → 部署 dashboard 8. `/dashboard` (无参数) → 全量刷新 + 部署 9. `/deploy`（在非 dashboard 项目）→ 通用部署 10. `/health doctools` → 文件卫生 + Git 状态 11. `/health` (在 ~/Work/zdwp) → 含 zdwp_health.py 12. `/groom doctools --check` → /pull + /audit 13. `/groom doctools` → 全流程含 /promote + /ship 14. `/promote dockit` → 独立可用 15. Streamlit 开发 → frontend skill 自动触发
