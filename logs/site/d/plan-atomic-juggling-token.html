# 计划：站点体系整合 — 统一 Navbar + GitHub Actions 自动部署 ## Context tianlizeng.cloud 下 28 子域名，之前工作已上线：stack / cmds / changelog / website / 6 Streamlit dashboard / 11 hydro。现状散乱： 1. **跳转孤立** — 每个站是独立孤岛，只有 `/services` 看板做统一入口；子站之间无互达链接，无"体系感" 2. **部署方式分裂** — 14 个已走 GitHub webhook 自动部署（dockit/cclog/11 hydro），9 个还在手工 `bash deploy.sh`：3 静态站（stack/cmds/cc-evolution）+ 6 动态服务（website 例外保留 deploy.sh，audiobook/oauth-proxy/repo-dashboard/hs-dashboard/auggie-dashboard） **用户决定**： - Q1 **A** — 每站顶部加统一 navbar（含 website、3 静态站、Streamlit 动态站） - Q2 **C** — 全部 8 个 webhook 化（除 website） - Q3 **统一 GitHub Actions**（"github actions 可以多用"） - SSH key **共用一把** - `deploy.sh` 保留作 fallback **预期**：commit push 即上线 + 所有站顶部一键跳转其他站。 --- ## 阶段 1：统一 Navbar 模板（本地改动，1h） ### 1.1 单一 navbar 来源 新建 `~/Dev/devtools/lib/templates/site-navbar.html`（和 `-streamlit.py`），各站 generate.py 或 Streamlit app 读它注入。 **navbar 内容**（固定 5-6 项，按重要性排序）： - `Home` → tianlizeng.cloud - `Services` → tianlizeng.cloud/services - `Stack` → stack.tianlizeng.cloud - `CC Docs` → cmds.tianlizeng.cloud - `Changelog` → changelog.tianlizeng.cloud - `Dashboard` → dashboard.tianlizeng.cloud 样式：固定顶栏 56px 高，亮色半透明，当前站高亮。 ### 1.2 注入各静态站（4 处） | 站 | 改动 | |---|---| | `~/Dev/stack/generate.py` | 读 navbar.html，注入到 `<body>` 首个元素 | | `~/Dev/cmds/generate.py` | 同上，在 index.html 和所有 <name>.html 都注入 | | `~/Dev/cc-evolution/generate.py` | 同上 | | `~/Dev/website/components/navbar.tsx` | 主站现有 navbar 加 4 个外链：Stack / CC Docs / Changelog / Dashboard | ### 1.3 Streamlit 动态站（5 处） `site-navbar-streamlit.py` 提供 `inject_navbar()` 函数：`st.markdown(open(path).read(), unsafe_allow_html=True)`。 改 5 个 app.py 顶部（都加一行）： - `~/Dev/audiobook/app/...` - `~/Dev/oauth-proxy/...` - `~/Dev/repo-dashboard/app.py` - `~/Dev/hs-dashboard/app.py` - `~/Dev/auggie-dashboard/app.py` --- ## 阶段 2：GitHub Actions 统一部署（8 个 repo，2h） ### 2.1 一次性准备 1. **生成共用 deploy SSH key**： ```bash ssh-keygen -t ed25519 -N "" -C "gha-deploy@tianlizeng.cloud" -f ~/.ssh/gha_deploy ``` 2. **公钥加 VPS**： ```bash cat ~/.ssh/gha_deploy.pub | ssh root@104.218.100.67 "cat >> ~/.ssh/authorized_keys" ``` 3. **批量注入 8 repo secret**（用 `gh secret set`，需 gh auth login 已配置）： ```bash for repo in stack cmds cc-evolution audiobook oauth-proxy repo-dashboard hs-dashboard auggie-dashboard; do gh secret set VPS_SSH_KEY -R zengtianli1/$repo < ~/.ssh/gha_deploy gh secret set VPS_HOST -R zengtianli1/$repo -b "104.218.100.67" done ``` ### 2.2 三类 workflow 模板 **Type A — 静态站（3 个）**：`stack / cmds / cc-evolution` `.github/workflows/deploy.yml`： ```yaml name: Deploy on: { push: { branches: [main] } } jobs: deploy: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-python@v5 with: { python-version: '3.11' } - run: pip install pyyaml - run: python3 generate.py - uses: webfactory/ssh-agent@v0.9.0 with: { ssh-private-key: '${{ secrets.VPS_SSH_KEY }}' } - run: rsync -avz --delete -e "ssh -o StrictHostKeyChecking=no" site/ root@${{ secrets.VPS_HOST }}:/var/www/<NAME>/ ``` **Type B — Streamlit dashboard（3 个）**：`repo-dashboard / hs-dashboard / auggie-dashboard` ```yaml - uses: webfactory/ssh-agent@v0.9.0 - run: rsync -avz --delete --exclude .venv --exclude __pycache__ ./ root@VPS:/opt/<NAME>/ - run: ssh root@VPS "systemctl restart <NAME>" ``` **Type C — FastAPI（2 个）**：`audiobook / oauth-proxy` 同 Type B，rsync 路径按 services.ts 的实际位置（/var/www/audiobook、/var/www/oauth-proxy）。 ### 2.3 保留 deploy.sh 每个 repo 的 `deploy.sh` **不删**，作为 fallback： - GHA 故障或离线时 `bash deploy.sh` 照常跑 - CLAUDE.md 里加一行："默认走 GHA；失败时用 deploy.sh 手工发" ### 2.4 首次部署验证 每个 repo commit + push 后： - GitHub Actions 页看 workflow 绿灯 - curl 对应域名确认更新 - 失败则 GHA logs 里看哪步挂（通常是 SSH key 或路径） --- ## 阶段 3：收尾记录（15 min） ### 3.1 Memory 新增 `reference_gha_deploy.md`（type: reference）： - 共用 deploy key 位置：`~/.ssh/gha_deploy` - 8 repo workflow 模板来源 - 新增站点加 webhook 化步骤（3 行） ### 3.2 Stack/CMDS 记录 - stack `projects.yaml` — 给 8 个新 webhook 化的站加标注 "GHA" - `cmds` 重新 generate（自动，不用手动） ### 3.3 HANDOFF.md 更新 --- ## 关键文件 | 路径 | 动作 | |---|---| | `~/Dev/devtools/lib/templates/site-navbar.html` | 新建 | | `~/Dev/devtools/lib/templates/site-navbar-streamlit.py` | 新建（Streamlit helper）| | `~/Dev/{stack,cmds,cc-evolution}/generate.py` | 加 navbar 注入 | | `~/Dev/website/components/navbar.tsx` | 加 4 外链 | | `~/Dev/{audiobook,oauth-proxy,repo-dashboard,hs-dashboard,auggie-dashboard}/**.py` | 加 `inject_navbar()` | | `~/.ssh/gha_deploy{,.pub}` | 新生成（本地）| | VPS `~/.ssh/authorized_keys` | 追加公钥 | | 8 个 repo 的 GitHub Secrets | `VPS_SSH_KEY` + `VPS_HOST` | | 8 个 repo 的 `.github/workflows/deploy.yml` | 新建（三类模板） | --- ## 可复用 - 现有 webhook_receiver.py 模式已证明部署链路（14 个 repo 跑了几个月）— Type B/C 抄它的 systemctl restart 逻辑 - `~/Dev/devtools/lib/tools/cf_api.py`（不动） - `~/Dev/cmds/generate.py` 里的 marked.js 模板风格 — navbar 复用同样亮色半透明 --- ## 验证 ```bash # 阶段 1 curl -sL https://stack.tianlizeng.cloud | grep -c "CC Docs" # ≥1 curl -sL https://cmds.tianlizeng.cloud | grep -c "Stack" # ≥1 # 浏览器打开每站顶部能看到 navbar，当前站高亮 # 阶段 2 # 每个 repo 改一个文件，commit push： cd ~/Dev/cmds && echo "# test" >> README.md && git add -A && git commit -m test && git push # 30s 后 curl cmds.tianlizeng.cloud — 应有更新 # GitHub Actions tab → 看 workflow 全绿 # 阶段 3 grep gha_deploy ~/.claude/projects/-Users-tianli-Dev/memory/MEMORY.md # ≥1 ``` --- ## 风险 & 回滚 | 风险 | 回滚 | |---|---| | GHA workflow 失败 | 本地 `bash deploy.sh`（fallback 保留） | | SSH key 泄漏 | 从 VPS `authorized_keys` 删 key + `gh secret delete` | | navbar 样式崩 | 回退 generate.py 的改动，恢复原 HTML | | website 被误 webhook 化 | 不动 website，明确排除 | --- ## 不做 -
