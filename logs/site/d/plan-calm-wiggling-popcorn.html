# 网站迁移 VPS + 搜索 + CMS 计划 ## Context 网站从 Vercel 迁移到自有 VPS（104.218.100.67），解锁动态功能。同时新增全文搜索和轻量 CMS 后台。域名保持 tianlizeng.cloud 根域名。 ## 现状 - **网站**: Next.js 15 + React 19 + Tailwind，当前部署在 Vercel - **VPS**: Ubuntu 22.04, 4C/8G, Node v22.22.1, 4.3GB 可用内存 - **Nginx**: 8443 端口，CF Origin 证书，所有服务统一反向代理 - **现有模式**: systemd service + Nginx reverse proxy + deploy.sh（dashboard/hydro-* 都这样） - **动态路由**: 仅 `/api/contact`（Resend 邮件）和 `/api/tools/[id]`（读 MD 文件），其余全 SSG - **环境变量**: RESEND_API_KEY + CONTACT_EMAIL（在 .env.local） ## 三大任务 --- ### 任务 A：VPS 部署（基础） #### A1. Next.js standalone 构建配置 修改 `~/Dev/website/next.config.mjs`，添加 `output: "standalone"`： ```js const nextConfig = { output: "standalone", // 新增 eslint: { ignoreDuringBuilds: true }, typescript: { ignoreBuildErrors: true }, images: { unoptimized: true }, }; ``` standalone 模式会在 `.next/standalone/` 生成自包含的 Node server，不需要 node_modules。 #### A2. 创建 deploy.sh `~/Dev/website/deploy.sh`，复用 repo-dashboard 的模式： ```bash #!/bin/bash set -e VPS="root@104.218.100.67" REMOTE_DIR="/opt/website" echo "📦 Building..." pnpm build echo "📦 Syncing to VPS..." rsync -avz --delete \ .next/standalone/ "$VPS:$REMOTE_DIR/" rsync -avz .next/static/ "$VPS:$REMOTE_DIR/.next/static/" rsync -avz public/ "$VPS:$REMOTE_DIR/public/" echo "🔄 Restarting..." ssh "$VPS" "systemctl restart website" echo "✅ Done! https://tianlizeng.cloud" ``` #### A3. VPS 上创建 systemd service SSH 到 VPS 创建 `/etc/systemd/system/website.service`： ```ini [Unit] Description=Personal Website (Next.js) After=network.target [Service] Type=simple WorkingDirectory=/opt/website Environment=NODE_ENV=production Environment=PORT=3000 Environment=HOSTNAME=127.0.0.1 EnvironmentFile=/etc/vps-secrets.env ExecStart=/usr/bin/node server.js Restart=always RestartSec=5 [Install] WantedBy=multi-user.target ``` 在 `/etc/vps-secrets.env` 中添加： ``` RESEND_API_KEY=re_bbznNopa_CSyURHqvSobRA38tasEB2UFM CONTACT_EMAIL=zengtianli1@gmail.com ``` #### A4. Nginx 配置 SSH 到 VPS 添加 Nginx server block（`/etc/nginx/conf.d/website.conf`）： ```nginx server { listen 8443 ssl; server_name tianlizeng.cloud; ssl_certificate /etc/nginx/ssl/origin.pem; ssl_certificate_key /etc/nginx/ssl/origin.key; location / { proxy_pass http://127.0.0.1:3000; proxy_http_version 1.1; proxy_set_header Upgrade $http_upgrade; proxy_set_header Connection "upgrade"; proxy_set_header Host $host; proxy_set_header X-Real-IP $remote_addr; proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for; proxy_set_header X-Forwarded-Proto $scheme; } } ``` #### A5. Cloudflare DNS 切换 用 cf_api.py 把 tianlizeng.cloud 的 A/CNAME 记录从 Vercel 改指 VPS IP（104.218.100.67），保持 proxied 状态。 #### A6. 删除 `/api/debug` 路由 `~/Dev/website/app/api/debug/route.ts` — 安全隐患，生产环境不需要。 --- ### 任务 B：全文搜索 #### B1. 创建 SQLite 数据库和索引脚本 新建 `~/Dev/website/scripts/build-search-index.py`： - 扫描 `content/` 下所有 .md 文件（blog、projects、research、resume、tools） - 解析 frontmatter（title、description、tags）+ 正文 - 写入 SQLite FTS5 表： ```sql CREATE VIRTUAL TABLE search_index USING fts5( slug, title, description, category, content, tokenize='unicode61' ); ``` - 输出到 `data/search.db`（gitignore） - 在 `deploy.sh` 的 build 步骤后自动运行，db 文件随部署上传 #### B2. 创建搜索 API 新建 `~/Dev/website/app/api/search/route.ts`： ```typescript // GET /api/search?q=水效评估 // 查询 SQLite FTS5，返回 JSON 结果（title, slug, category, snippet） ``` 使用 `better-sqlite3` 包（Node 原生 SQLite 绑定，性能好）。 #### B3. 创建搜索页面 新建 `~/Dev/website/app/search/page.tsx`： - 搜索框 + 即时结果列表 - 调用 `/api/search?q=xxx` - 结果按 category 分组（论文 / 项目 / 博客 / 工具） - 高亮匹配关键词 #### B4. 全局导航加搜索入口 修改 `~/Dev/website/content/global/navbar.md` 或导航组件，加搜索图标/入口。 --- ### 任务 C：轻量 CMS 后台 #### C1. CMS API 路由 新建 `~/Dev/website/app/api/admin/` 下的 API： ``` POST /api/admin/posts — 创建博客文章（写 .md 文件到 content/blog/） PUT /api/admin/posts/[slug] — 编辑博客文章 DELETE /api/admin/posts/[slug] — 删除博客文章 POST /api/admin/upload — 上传图片到 public/images/ ``` 核心逻辑：接收 JSON（title, content, tags）→ 生成 frontmatter → 写入 .md 文件 → 触发 `next build` 的 ISR 或按需重建。 **注意**：standalone 模式下写文件后需要重建才能看到新页面。两种方案： - **方案 1**: CMS 写文件后自动触发 `pnpm build && systemctl restart website`（简单但有几秒停机） - **方案 2**: 改用 ISR（Incremental Static Regeneration），动态页面按需重建（推荐，无停机） 选方案 2：在动态路由（blog/[slug]）中加 `revalidate` 配置。 #### C2. CMS 前端页面 新建 `~/Dev/website/app/admin/` 下： ``` /admin — 文章列表（标题、日期、状态） /admin/new — 新建文章（Markdown 编辑器 + 实时预览） /admin/edit/[slug] — 编辑文章 ``` Markdown 编辑器用轻量方案：`<textarea>` + 实时预览（remark 渲染），不引入重型富文本编辑器。 #### C3. Cloudflare Access 保护 用 cf_api.py 给 `tianlizeng.cloud/admin*` 路径添加 CF Access 策略： - 允许邮箱：zengtianli1@gmail.com - 验证方式：邮箱验证码 #### C4. 图片上传 CMS 编辑器支持拖拽/粘贴上传图片： - 前端：拖拽区域 → FormData → POST /api/admin/upload - 后端：保存到 `public/images/blog/` → 返回 URL - 插入到编辑器中的 Markdown --- ## 执行顺序 ``` Phase 1: VPS 部署 (A1-A6) ├── A1 修改 next.config.mjs (standalone) ├── A2 创建 deploy.sh ├── A3 VPS: systemd service ├── A4 VPS: Nginx config ├── A5 CF DNS 切换 └── A6 删除 debug 路由 Phase 2: 全文搜索 (B1-B4) ├── B1 索引脚本 + SQLite ├── B2 搜索 API ├── B3 搜索页面 └── B4 导航加搜索入口 Phase 3: CMS 后台 (C1-C4) ├── C1 API 路由 ├── C2 前端页面 ├── C3 CF Access └── C4 图片上传 ``` Phase 1 必须先完成（后面两个都依赖 VPS 运行）。Phase 2 和 3 可以并行。 ## 新增依赖 - `better-sqlite3` — Node 原生 SQLite 绑定（搜索用） - 无其他新依赖（CMS 用现有的 React + API Routes） ## 关键文件清单 | 文件 | 操作 | 任务 | |------|------|------| | `next.config.mjs` | 修改 | A1 | | `deploy.sh` | 新建 | A2 | | `app/api/debug/route.ts` | 删除 | A6 | | `scripts/build-search-index.py` | 新建 | B1 | | `app/api/search/route.ts` | 新建 | B2 | | `app/search/page.tsx` | 新建 | B3 | | `app/api/admin/posts/route.ts` | 新建 | C1 | | `app/api/admin/posts/[slug]/route.ts` | 新建 | C1 | | `app/api/admin/upload/route.ts` | 新建 | C1 | | `app/admin/page.tsx` | 新建 | C2 | | `app/admin/new/page.tsx` | 新建 | C2 | | `app/admin/edit/[slug]/page.tsx` | 新建 | C2 | | VPS: `/etc/systemd/system/website.service` | 新建 | A3 | | VPS: `/etc/nginx/conf.d/website.conf` | 新建 | A4 | ## 验证 ### Phase 1 验证 1. 本地 `pnpm build` 成功，`.next/standalone/` 生成 2. `bash deploy.sh` 部署到 VPS 3. `curl -I https://tianlizeng.cloud` 返回 200 4. 联系表单发送测试邮件成功 ### Phase 2 验证 1. `python3 scripts/build-search-index.py` 生成 `data/search.db` 2. `/search` 页面可搜索，输入"水效"能找到对应项目和软著 3. 结果按类别分组，点击可跳转 ### Phase 3 验证 1. `/admin` 页面被 CF Access 拦截，邮箱验证后可访问 2. 新建文章 → 保存 → 文章在 `/blog/[slug]` 可访问 3. 上传图片 → 图片在文章中正常显示
