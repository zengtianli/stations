# Plan · ~/Dev 深度重构 v0.4 · 根下漂移清剿 + 工具层落地 ## Context `~/Dev/` 在 2026-04-20 做过分层重构（`stations/` / `labs/` / `content/` / `tools/` / `migrated/` / `_archive/`），但仍有 **3 个根下漂移者** 绕过了分层协议： - `~/Dev/docs/`（独立 repo，只含 `health/` 子目录 + `.git`，和 `stations/docs` 完全不同） - `~/Dev/ops-console/`（独立 repo，只含 `data/` + `.git`，是 `stations/ops-console` 重构前的空壳） - `~/Dev/ticket-reminder/`（2026-04-24 新建的 GitHub repo，刚起步，未定位） 同时，2026-04-24 立了 `paths.yaml` SSOT + `paths.py rewrite-dead` 工具，但： - 现有 `/station-promote` skill **不消费** `paths.yaml` 的 `migrations`（retro doc 自承的缺口） - `~/Dev/tools/configs/playbooks/` 下还没有 `paths.md`，路径搬迁 SOP 散在 CLAUDE.md / retro doc 各处 **本次目标**：3 个漂移者各就各位、零新死链、把搬迁编排固化成可复用 skill + playbook。未来每次搬目录靠 `/refactor-dir <old> <new>` 一键跑完（preflight → 加 migration → mv → rewrite-dead → build-const → scan-dead 验证）。 **用户已决策**（AskUserQuestion 锁定）： 1. `~/Dev/Work/` 保持不动（业务目录特例，登记 paths.yaml 让 audit 不报噪即可） 2. 根下 `docs/` + `ops-console/` → 两个都归 `_archive/` 3. `ticket-reminder/` → `labs/ticket-reminder/`（实验/孵化） 4. 工具方案：立新 `/refactor-dir` skill + 写 `playbooks/paths.md` **范围外**（明确不做）： - ❌ `migrated/hydro-*` 10 项（HANDOFF 约定 2026-05-04 归档，今天 4-24，到点再做） - ❌ labs/ 内部清理（llm-finetune / mindmap 还活着，全留） - ❌ 改 `/station-promote`（语义不同，新 skill 和它共存） --- ## Deliverables ### 1. 目录搬迁（4 项） | 源 | 目标 | 理由 | |---|---|---| | `~/Dev/docs/` | `~/Dev/_archive/docs-health-2026-04/` | 独立 repo 空壳，只剩 health 子目录，和 stations/docs 完全冲突命名 | | `~/Dev/ops-console/` | `~/Dev/_archive/ops-console-stub-2026-04/` | 独立 repo 空壳，data/ 是 stations/ops-console 重构前残留 | | `~/Dev/ticket-reminder/` | `~/Dev/labs/ticket-reminder/` | 新 repo，未上线，归实验层 | | `~/Dev/_deadlinks-remaining.md` | `~/Dev/_archive/deadlinks-reports/_deadlinks-remaining-2026-04-24.md` | 轮 2 清理产物，留档 | ### 2. paths.yaml 更新（`~/Dev/paths.yaml` 手写 SSOT） **新增 migrations（7 条）**（双写 `~/` + `/Users/tianli/Dev/`）： ```yaml - {from: ~/Dev/ticket-reminder, to: ~/Dev/labs/ticket-reminder} - {from: /Users/tianli/Dev/ticket-reminder, to: /Users/tianli/Dev/labs/ticket-reminder} # docs / ops-console 归 _archive 的 migration（只映射到 archive 目录） - {from: ~/Dev/docs/health, to: ~/Dev/_archive/docs-health-2026-04/health} - {from: /Users/tianli/Dev/docs/health, to: /Users/tianli/Dev/_archive/docs-health-2026-04/health} - {from: ~/Dev/ops-console/data, to: ~/Dev/_archive/ops-console-stub-2026-04/data} - {from: /Users/tianli/Dev/ops-console/data, to: /Users/tianli/Dev/_archive/ops-console-stub-2026-04/data} ``` ⚠️ **不加** `~/Dev/docs` / `~/Dev/ops-console` 本身的 migration —— 因为现有 migrations 已经有 `~/Dev/docs → ~/Dev/stations/docs` 和 `~/Dev/ops-console → ~/Dev/stations/ops-console`（这是历史正确映射，不能覆盖）。根下空壳归档只移动具体子目录。 **新增 registered ids（3 条）**： ```yaml dirs: labs: ~/Dev/labs content: ~/Dev/content archive: ~/Dev/_archive files: paths_yaml: ~/Dev/paths.yaml paths_py: ~/Dev/devtools/lib/tools/paths.py ``` **新增 allow_missing（1 条）**： ```yaml - path: ~/Dev/Work reason: 用户业务目录（水利项目 / 培训 / 投标），不是代码工作区；替换旧的 ~/Work 条目 added: 2026-04-24 ``` （`~/Dev/Work` 实际存在，此处用 allow_missing 的语义**其实不对** —— 正确做法是把它登记为 registered id。修改方案：加 `dirs.work_biz: ~/Dev/Work`，并删掉旧的 `~/Work` allow_missing 那条，因为 `~/Work` 根本不存在，`~/Dev/Work` 才是真实的业务目录。） **修订**： ```yaml dirs: work_biz: ~/Dev/Work # 业务目录（非代码），audit 识别它存在即可 # allow_missing 中：删 ~/Work 那条（已过时，业务目录搬到 ~/Dev/Work） ``` ### 3. 新立 skill：`/refactor-dir <old> <new>` **文件**：`~/Dev/tools/cc-configs/commands/refactor-dir.md`（markdown skill） **参数**： - `<old>` 绝对路径或 `~/Dev/...` 形式 - `<new>` 目标路径 - `--dry` 只打印计划不执行 - `--no-rewrite` 跳过 rewrite-dead（搬归档目录时可能用） - `--reason <text>` migration 条目的注释 **编排 8 步**： ``` 1. Preflight : 验证 <old> 存在、<new> 不存在、<old> 不是 git-clean 的未提交状态 2. Grep : 扫 cc-configs/ configs/ devtools/ stations/docs/ 里的 <old> 引用次数（预警） 3. paths.yaml : 追加 migration 条目（双写 ~/ + /Users/tianli/） 4. mv : mv <old> <new> 5. rewrite-dead: python3 paths.py rewrite-dead --dry-run → 看影响 → 实跑（除非 --no-rewrite） 6. build-const : python3 paths.py build-const -w（若涉及 registered id 变更） 7. scan-dead : python3 paths.py scan-dead --strict（验证 0 新死链） 8. Summary : 打印"接下来要 commit 的 repo 列表"（~/Dev/paths.yaml 本身 + 影响的各 repo） ``` **失败回滚**： - Step 4 失败 → 删 paths.yaml 新加的 migration 条目 - Step 5 失败 → `mv <new> <old>` 反向 + paths.yaml 同上回滚 - Step 7 失败（发现新死链）→ 不自动回滚（新死链可能是历史债），打印详情让用户判断 **不做**： - 不 touch GitHub remote（搬目录不改 repo 归属，独立 repo 搬到 _archive/ 仍保留 origin） - 不自动 commit（让用户审阅 diff 后决定） - 不动 symlink / systemd / nginx（SSOT 在代码层，不耦合运行时） ### 4. 新立 playbook：`paths.md` **文件**：`~/Dev/tools/configs/playbooks/paths.md` 按 META.md §3 的 8 节骨架： - **定位**：路径域任务 = 路径 SSOT 管理 + 搬迁 + 死链清理 - **入场判断**：trigger words = 搬目录 / 搬 repo / 归档项目 / 改 paths.yaml / 清死链 / refactor-dir - **编排图**：ASCII 流程图（preflight → refactor-dir → commit） - **复用清单**：paths.py 4 件套 + /refactor-dir + /preflight + git + grep - **改动边界矩阵**：paths.yaml 手写唯一改点 / paths_const.py 禁改（派生）/ migrations 条目顺序 - **踩坑 anti-patterns**： - rewrite-dead 的 lookahead 漏 backtick（已修，记录防再犯） - 父路径误伤子路径（migrations 顺序要具体在前） - 同一搬迁必须双写 `~/` 和 `/Users/tianli/` 两个 from - allow_missing 用 startswith 前缀匹配，一条可盖多 ref - **文件索引**：paths.yaml / paths.py / paths_const.py / _deadlinks-remaining.md - **扩展机制**：新增 registered id / 新增 migration / 新立 shell/ts paths_const 同时更新 `META.md §5` 加一行：`| paths | paths.md | 搬目录 / 归档项目 / 改 paths.yaml / 清死链 / refactor-dir | ✅ 已立 |` ### 5. CLAUDE.md 更新（`~/Dev/CLAUDE.md`） 在「路径注册表 SSOT」节末尾追加： - `/refactor-dir` 使用方式 1 行 - playbook 位置指引：搬迁工作流见 `~/Dev/tools/configs/playbooks/paths.md` --- ## Critical Files **新建**： - `~/Dev/tools/cc-configs/commands/refactor-dir.md`（新 skill） - `~/Dev/tools/configs/playbooks/paths.md`（新 playbook） - `~/Dev/_archive/docs-health-2026-04/`（搬迁目标，mv 时自动创建父目录） - `~/Dev/_archive/ops-console-stub-2026-04/`（同上） - `~/Dev/_archive/deadlinks-reports/`（同上） **修改**： - `~/Dev/paths.yaml`（加 migrations / dirs / 删过时 allow_missing） - `~/Dev/devtools/lib/paths_const.py`（通过 `paths.py build-const -w` 自动刷，不手改） - `~/Dev/CLAUDE.md`（追加 refactor-dir 指引） - `~/Dev/tools/configs/playbooks/META.md`（§5 加 paths 索引行） - `~/Dev/devtools/scripts/refactor-dir.sh`（可选：skill 若想要 bash wrapper；或纯 prompt-based） **删除**： - `~/Dev/_deadlinks-remaining.md`（搬到 `_archive/deadlinks-reports/` 后原位删） **移动**： - `~/Dev/docs/` → `~/Dev/_archive/docs-health-2026-04/` - `~/Dev/ops-console/` → `~/Dev/_archive/ops-console-stub-2026-04/` - `~/Dev/ticket-reminder/` → `~/Dev/labs/ticket-reminder/` --- ##
