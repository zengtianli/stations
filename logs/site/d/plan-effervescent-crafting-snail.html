# Plan: 创建 cc-harness 和 cc-context 两个独立工具仓库 ## Context 用户需要两个独立 GitHub 仓库，各自是可运行的 CLI 工具： 1. **cc-harness** — CC 配置质量审计（检测 CLAUDE.md/skills/hooks/rules 是否健康，输出六维度评分） 2. **cc-context** — 上下文工程工具包（token 监控 + compact 前自动存档 + 会话健康检查） 两者的共同目标：让 CC 表现更好、对话不崩。 ### 已有资源 - `~/.agents/skills/health/SKILL.md` — 现有审计逻辑（bash 数据收集 + 两个 subagent 分析），可直接复用 Step 1 的 bash 脚本 - `~/Dev/cclog/` — 已有 .jsonl 解析器（parser.py 278行），token 统计逻辑可复用 - .jsonl 文件包含完整的 token 使用数据（input/output/cache_read/cache_creation） --- ## 仓库 1: cc-harness **位置**：`~/Dev/cc-harness/` **远程**：`zengtianli/cc-harness` (public) **定位**：Claude Code 配置质量审计工具，不依赖 CC 也能跑基础检查 ### 目录结构 ``` cc-harness/ ├── README.md / README_CN.md ├── LICENSE (MIT) ├── harness.py ← CLI 入口（python3 harness.py [project_path]） ├── collectors/ │ ├── __init__.py │ ├── config.py ← 收集 CLAUDE.md、settings.json、rules/ │ ├── skills.py ← 收集 skill 清单、frontmatter、安全扫描 │ ├── hooks.py ← 收集 hooks 配置、验证 schema │ └── metrics.py ← 源文件数、贡献者、CI、启动 token 预算 ├── analyzers/ │ ├── __init__.py │ ├── context.py ← D1: CLAUDE.md 质量、重复/冲突检测、token 预算 │ ├── hooks.py ← D2: hooks 覆盖率、schema 合法性 │ ├── agents.py ← D3: skill 数量、职责重叠、描述符质量 │ ├── verification.py ← D4: done-conditions 覆盖率 │ ├── session.py ← D5: Compact Instructions、Handoff、方案模板 │ └── structure.py ← D6: 孤立文件、引用链、命名规范 ├── reporters/ │ ├── __init__.py │ ├── markdown.py ← 输出 markdown 报告（Critical/Structural/Incremental） │ └── scorecard.py ← 输出六维度评分表 ├── security/ │ ├── __init__.py │ └── scanner.py ← skill 安全扫描（prompt injection、凭证泄露等 6 类） └── tests/ ├── fixtures/ ← 模拟的 CLAUDE.md、settings.json、SKILL.md ├── test_collectors.py ├── test_analyzers.py └── test_security.py ``` ### 核心功能 ```bash # 基础审计（纯静态分析，不需要 CC） python3 harness.py ~/Work/zdwp/ # 输出示例： # === Harness Score Card — 2026-03-31 === # D1 Context 7/10 CLAUDE.md 有 3 处重复规则 # D2 Hooks 3/10 无 PostToolUse hooks # D3 Agents 6/10 2 个 skill 描述过长 # D4 验证 4/10 缺少 Verification 节 # D5 会话 5/10 无 Compact Instructions # D6 结构 8/10 1 个嵌套 CLAUDE.md # 总分 5.5/10 # # 🔴 Critical (2): 硬编码 API Token、settings.json 未 gitignore # 🟡 Structural (5): ... # 🟢 Incremental (3): ... # JSON 输出（可被其他工具消费） python3 harness.py ~/Work/zdwp/ --json # 只跑安全扫描 python3 harness.py ~/Work/zdwp/ --security-only ``` ### 评分逻辑（每维度 0-10） 每个维度有一组检查项，每项 pass=满分/fail=0，最终取加权平均。例如 D1： - Token 占用 <10%? (2分) - 全局/本地无重复规则? (2分) - 无嵌套 CLAUDE.md? (2分) - MEMORY.md 存在且更新? (1.5分) - Compact Instructions 存在? (1.5分) - 无硬编码凭证? (1分) ### 分级逻辑（自动判定 Simple/Standard/Complex） ```python if source_files < 500 and contributors <= 1 and ci_workflows == 0: tier = "simple" elif source_files > 5000 and contributors > 3 and ci_workflows > 0: tier = "complex" else: tier = "standard" ``` Simple 不检查 hooks/rules/verification，Standard 检查基础项，Complex 全检查。 --- ## 仓库 2: cc-context **位置**：`~/Dev/cc-context/` **远程**：`zengtianli/cc-context` (public) **定位**：Claude Code 上下文工程工具包，管理 token 消耗和会话健康 ### 目录结构 ``` cc-context/ ├── README.md / README_CN.md ├── LICENSE (MIT) ├── context.py ← CLI 入口 ├── monitor/ │ ├── __init__.py │ ├── token_stats.py ← 解析 .jsonl，统计 token 消耗分布 │ └── cost_tracker.py ← 估算 API 成本（按模型定价） ├── snapshot/ │ ├── __init__.py │ ├── auto_save.py ← compact 前自动保存上下文到文件 │ └── restore.py ← compact 后从存档恢复关键信息 ├── health/ │ ├── __init__.py │ ├── session_check.py ← 分析会话：轮数、compact 次数、token 浪费模式 │ └── patterns.py ← 检测反模式（重复搜索、大文件读取、应拆分信号） ├── hooks/ │ ├── pre_compact.sh ← hook 脚本：compact 前自动调用 auto_save │ └── install.sh ← 一键安装 hook 到 settings.json └── tests/ ├── fixtures/ ← 模拟的 .jsonl 会话文件 ├── test_monitor.py ├── test_snapshot.py └── test_health.py ``` ### 核心功能 #### 1. Token 监控 ```bash # 分析最近一次会话的 token 消耗 python3 context.py monitor # 输出： # === Session Token Report === # Duration: 2h 15m (278 turns) # Input: 16,159 tokens # Output: 64,439 tokens # Cache Read: 20.7M tokens (cache hit rate: 93.4%) # Cache Write: 1.4M tokens # Est. Cost: $0.82 # # Top 5 Token Consumers: # 1. Read tool (43%) — 读了 87 个文件 # 2. Bash tool (22%) — 执行了 45 个命令 # 3. Agent tool (18%) — 3 个 subagent # 4. Grep tool (9%) — 23 次搜索 # 5. Edit tool (8%) — 15 次编辑 # # ⚠️ 建议：3 次重复搜索同一关键词，考虑用 augment # 分析指定会话 python3 context.py monitor --session 92da79d7 # 分析所有会话，输出趋势 python3 context.py monitor --all --json ``` #### 2. Compact 前自动存档 ```bash # 手动触发存档（保存当前会话的关键信息） python3 context.py snapshot save # 输出文件：~/.claude/snapshots/{session-id}-{timestamp}.md # 内容： # - 当前任务进度 # - 已修改文件列表（从 tool_use 提取） # - 关键决策（从对话中提取） # - Token 消耗摘要 # 恢复（compact 后读取） python3 context.py snapshot restore # 安装为 hook（compact 前自动触发） python3 context.py hooks install ``` 存档文件格式： ```markdown # Context Snapshot — 2026-03-31 09:15 ## Session: 92da79d7 ### 修改的文件 - ~/Learn/claude-code/theory/Layer1-长期上下文.md (edited) - ~/Learn/claude-code/README.md (created) - ~/Learn/README.md (edited) ### Tool 使用统计 - Read: 45 次 - Edit: 15 次 - Bash: 12 次 - Agent: 3 次 ### Token 消耗 - Total input: 16K, output: 64K - Cache hit rate: 93.4% ``` #### 3. 会话健康检查 ```bash # 检查最近会话的健康度 python3 context.py health # 输出： # === Session Health Report === # ✅ Duration: 2h 15m (within range) # ⚠️ Turns: 278 (>100, consider splitting) # ⚠️ Compact events: 3 (high, context pressure) # ❌ Repeated searches: 5 patterns detected # ✅ Cache hit rate: 93.4% (excellent) # # Recommendations: # 1. 考虑拆分会话（已超 100 轮） # 2. 3 次重复搜索 "CLAUDE.md" — 考虑用 memory 记住位置 # 3. 读取了 5 个 >500 行的文件 — 考虑用 offset/limit 精确读取 # 分析所有项目的会话 python3 context.py health --all ``` ### 复用 cclog 解析逻辑 `~/Dev/cclog/src/cclog/parser.py` 已有完善的 .jsonl 解析器，直接复用： - `parse_session()` — 解析会话文件 - token 字段提取逻辑 - 跳过无关条目（file-history-snapshot, progress, queue-operation） --- ## 执行步骤 ### Phase 1: cc-harness（先做，因为有现成 skill 可参考） 1. `mkdir ~/Dev/cc-harness && cd ~/Dev/cc-harness && git init` 2. 从 health SKILL.md Step 1 的 bash 脚本提取 collectors 逻辑 3. 实现 6 个 analyzer（D1-D6），每个输出 findings + score 4. 实现 security scanner（6 类检查） 5. 实现 markdown reporter + scorecard 6. 写 CLI 入口 `harness.py` 7. 写 tests（用 fixtures 模拟配置） 8. 写 README（中英双语） 9. 推送 GitHub ### Phase 2: cc-context（依赖 cclog parser 理解） 1. `mkdir ~/Dev/cc-context && cd ~/Dev/cc-context && git init` 2. 从 cclog parser.py 复用 .jsonl 解析逻辑 3. 实现 token_stats（按工具分类统计） 4. 实现 auto_save（提取修改文件列表 + token 摘要） 5. 实现 session_check + patterns（反模式检测） 6. 写 hook 安装脚本 7. 写 tests 8. 写 README 9. 推送 GitHub ### Phase 3: 关联 -
