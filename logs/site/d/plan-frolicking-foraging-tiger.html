# Plan · 立 `~/Dev/paths.yaml` 路径注册表（路线 A 薄壳） ## Context **为什么做**：用户反复经历"重构 → 死链"（目录搬迁、repo 改名后 CLAUDE.md / playbooks / 脚本里裸路径全部要手工 sed）。已有零散 SSOT（`repo-map.json` / `menus/` / `station_path.py`）覆盖不全，CLAUDE.md / playbooks / slash command / 脚本里仍有大量裸绝对路径。 **用户在前置讨论中已决定**： - `~/Work` → `~/Dev` 已由用户手工完成搬迁（无需脚本处理） - 路线 **A · 薄壳 paths.yaml**（1-2h 立项，可无缝演进到 B） - 覆盖：本地绝对路径 + VPS 路径 + 域名/URL + 环境变量名（**4 类全要**） - 文档重写策略：**新写的用 map，旧文件不动**，靠 audit 慢慢迁 - 脚本路径：**代码生成 `paths_const.py`**（类 services.ts 派生模式） - 审计：**`paths.py audit` 手动跑**（不进 pre-commit，不上 CI） **痛点命中**：立项后，新写 CLAUDE.md / playbook / 脚本时优先 reference map，下次重构只改 map 一行 + 跑 `paths sync` 刷派生产物。旧文件随 audit 报告慢慢回归。 --- ## 产物清单 ### 新建 3 个文件 | 路径 | 作用 | 幂等来源 | |---|---|---| | `~/Dev/paths.yaml` | SSOT 手写（唯一手改点） | — | | `~/Dev/devtools/lib/tools/paths.py` | CLI：resolve / audit / build-const / sync | — | | `~/Dev/devtools/lib/paths_const.py` | AUTO-GENERATED Python 常量模块 | `paths.py build-const` | ### 可选扩展（v0.2，本次不做） - `~/Dev/devtools/lib/paths_const.sh` — bash `export` 版本，供 shell 脚本 source - `~/Dev/devtools/lib/paths_const.ts` — TS 版本，供 Next.js 消费 - pre-commit hook 接入 --- ## paths.yaml schema（基于扫描 Top 30 裸路径归纳） 沿用 `menus/` 的分组思维，但扁平化（单文件）。按**用途域**分 7 个顶级键： ```yaml # ~/Dev/paths.yaml # SSOT · 路径注册表 · 手写 # 派生产物：paths_const.py (由 `paths.py build-const -w` 生成) version: 1 updated: 2026-04-24 roots: dev: ~/Dev home: ~ claude: ~/.claude # 大目录（被脚本/文档频繁引用） dirs: stations: ~/Dev/stations devtools: ~/Dev/devtools devtools_lib: ~/Dev/devtools/lib devtools_lib_tools: ~/Dev/devtools/lib/tools devtools_lib_templates: ~/Dev/devtools/lib/templates tools_configs: ~/Dev/tools/configs tools_cc_configs: ~/Dev/tools/cc-configs menus_ssot: ~/Dev/tools/configs/menus playbooks: ~/Dev/tools/configs/playbooks nginx_out: ~/Dev/tools/configs/nginx/out web_stack: ~/Dev/stations/web-stack web_stack_services: ~/Dev/stations/web-stack/services web_stack_apps: ~/Dev/stations/web-stack/apps # 高频被引用的具体文件 files: claude_md_dev: ~/Dev/CLAUDE.md claude_md_global: ~/.claude/CLAUDE.md handoff_dev: ~/Dev/HANDOFF.md repo_map: ~/Dev/tools/configs/repo-map.json menus_py: ~/Dev/devtools/lib/tools/menus.py personal_env: ~/.personal_env playbook_meta: ~/Dev/tools/configs/playbooks/META.md # VPS vps: ssh: root@104.218.100.67 ip: 104.218.100.67 www_root: /var/www nginx_conf: /etc/nginx nginx_sites: /etc/nginx/sites-enabled systemd: /etc/systemd/system # 域名 domains: apex: tianlizeng.cloud # 环境变量名（只登记存在性，不含 value） env_vars: - CF_API_TOKEN - CF_ACCOUNT_ID - CF_ZONE_ID - GH_TOKEN - GH_PAGES_TOKEN - SMTP_PASSWORD - OPENAI_API_KEY - ANTHROPIC_API_KEY ``` **裁剪原则**：扫描找到 30 条，保留 ~25 条最高频。`hydro-*` 10 个服务名不入 paths.yaml（那是 services domain，属于 menus/）。`.tianlizeng.cloud` 子域不入（由 menus/entities/subdomains.yaml 管）。**paths.yaml 只管路径 / VPS / env 存在性，不管服务实体**。 --- ## paths.py CLI 设计（复用 menus.py / repo_map_gen.py pattern） ### 子命令 ```bash # 查询 python3 paths.py resolve <id> # 打印单条，如 resolve menus_ssot → /Users/tianli/Dev/tools/configs/menus python3 paths.py resolve <id> --expand # 展开 ~ 为绝对路径 python3 paths.py list # 打印全部 id + 路径 # 派生产物 python3 paths.py build-const # 打印预览 paths_const.py python3 paths.py build-const -w # 写入 ~/Dev/devtools/lib/paths_const.py # 审计 python3 paths.py audit # 扫漂移 + 硬编码，输出报告 python3 paths.py audit --strict # 有漂移则 exit 1 ``` ### audit 检查的 4 项 1. **路径存在性**：`paths.yaml` 里声明的每条 `dirs:` / `files:` 是否真实存在（`missing` 列表） 2. **硬编码扫描**：扫 `~/Dev/CLAUDE.md` + `~/.claude/CLAUDE.md` + `~/Dev/tools/configs/playbooks/*.md` + `~/Dev/tools/cc-configs/commands/*.md` 里的裸绝对路径，报告"应入 map 但未入"的候选（`unregistered` 列表，按出现频次排序） 3. **派生产物陈旧**：`paths_const.py` 和当前 `paths.yaml` 是否一致（对比 hash / 逐条比对） 4. **env_vars 存在性**：`~/.personal_env` 里声明的 env_vars 是否都设了（`missing_env` 列表） 输出格式参考 `menus.py audit`（GREEN/RED/YELLOW 彩色 + 分类报告）。 ### build-const 生成的 paths_const.py ```python # AUTO-GENERATED from ~/Dev/paths.yaml — do not edit # Regenerate: python3 ~/Dev/devtools/lib/tools/paths.py build-const -w from pathlib import Path DEV_ROOT = Path("/Users/tianli/Dev") HOME = Path("/Users/tianli") CLAUDE = Path("/Users/tianli/.claude") # dirs STATIONS = Path("/Users/tianli/Dev/stations") DEVTOOLS = Path("/Users/tianli/Dev/devtools") DEVTOOLS_LIB = Path("/Users/tianli/Dev/devtools/lib") # ... 其余条目 # files CLAUDE_MD_DEV = Path("/Users/tianli/Dev/CLAUDE.md") # ... # VPS (strings, 非 Path) VPS_SSH = "root@104.218.100.67" VPS_IP = "104.218.100.67" # ... # env var names (strings) ENV_VARS = [ "CF_API_TOKEN", # ... ] ``` --- ## 实施步骤 1. **写 paths.yaml**（手工，~15min） - 基于探索报告的 Top 30，裁到 ~25 条 - 验证每条 `dirs:` / `files:` 现实存在 2. **写 paths.py**（~45min） - argparse 子命令布局仿 `menus.py` 第 1715-1761 行 - `resolve` / `list` / `build-const` / `audit` 4 个子命令 - audit 的硬编码扫描用 `re.finditer(r'(/Users/tianli/\S+|~/[\w./-]+)', content)` - 彩色输出用同样的 GREEN/RED/YELLOW ANSI 常量 3. **生成 paths_const.py**（1 条命令） - `python3 ~/Dev/devtools/lib/tools/paths.py build-const -w` 4. **跑首次 audit** - `python3 ~/Dev/devtools/lib/tools/paths.py audit` - 预期：`missing` 为空；`unregistered` 报一堆候选（正常，后续慢慢迁）；`missing_env` 按 `~/.personal_env` 实际状况 5. **更新 ~/Dev/CLAUDE.md** - 新增一节「路径注册表 SSOT」，记录 paths.yaml + paths.py 用法 - 插入位置：「站群菜单/导航 SSOT」章节之后（保持"Xxx SSOT"聚类） 6. **更新 HANDOFF.md** - 把 `~/Dev/HANDOFF.md` 的"路径注册表立项"状态从"用户起意"改为"v0.1 已立，观察期 2 周" - 记下"下次升级到路线 B 的触发条件" --- ## 关键文件路径（施工时用） - **新建**：`~/Dev/paths.yaml` - **新建**：`~/Dev/devtools/lib/tools/paths.py` - **新建（AUTO-GENERATED）**：`~/Dev/devtools/lib/paths_const.py` - **参考实现**： - `~/Dev/devtools/lib/tools/menus.py`（CLI 布局 + audit pattern + 彩色输出） - `~/Dev/devtools/lib/tools/repo_map_gen.py`（scan + merge-aware 思路） - `~/Dev/devtools/lib/station_path.py`（路径解析） - **修改**：`~/Dev/CLAUDE.md`（新增一节文档） - **修改**：`~/Dev/HANDOFF.md`（状态更新） --- ## 验证（end-to-end） ```bash # 1. 安装后自检 cd ~/Dev python3 devtools/lib/tools/paths.py list | head -30 python3 devtools/lib/tools/paths.py resolve menus_ssot --expand # 应输出 /Users/tianli/Dev/tools/configs/menus # 2. 派生产物 python3 devtools/lib/tools/paths.py build-const -w python3 -c "from devtools.lib.paths_const import MENUS_SSOT; print(MENUS_SSOT)" # 3. audit python3 devtools/lib/tools/paths.py audit # 预期：paths.yaml 里所有 dirs/files 都 exist；unregistered 报候选列表；paths_const.py 与 paths.yaml 一致 # 4. 演进测试（假装改一条路径） # （手工把 paths.yaml 里某条改成错的 → audit 应报 missing → 改回来 + build-const -w → audit 绿） ``` --- ## 明确不做（范围界限） - ❌ 不自动 sed 旧文件（保守策略，靠 audit 报告慢慢迁） - ❌ 不接 pre-commit（用户选了"手动跑"） - ❌ 不碰 `hydro-*` 服务名 / 子域（那属于 menus/ domain） - ❌ 不改 `station_path.py` / `repo_map_gen.py`（共存，不合并） - ❌ 不立 `paths_const.sh` / `paths_const.ts`（v0.2 再说，等实际有 shell/TS 脚本 import 需求时） - ❌ 不做路线 B 的 entities+relations 拆分（先用 A 验证痛点） --- ## 回滚预案 所有产物都在 `~/Dev/devtools/lib/` 下，`rm paths.yaml paths_const.py tools/paths.py` 即可完整回滚。CLAUDE.md / HANDOFF.md 的改动用 git 回退。不涉及任何 VPS / 外部系统修改。
