# Plan · HS Dashboard 功能开关化 + Raycast/HS 职责分明 ## Context（为什么做） 当前 `~/Dev/hs-dashboard`（线上 https://hammerspoon.tianlizeng.cloud，VPS 部署）只是**只读展示面板**： - 数据源：`data/hs_config.json`，由 `~/.hammerspoon/lib/export_config.lua` 在 HS 启动时导出 - 展示：快捷键表 + 被动功能 + 设置 + 架构图 - **不区分 Raycast / Hammerspoon**，实际上该 dashboard 只含 HS 数据，压根没有 Raycast - **没有任何交互**——想开关某个功能，得去编辑 `~/.hammerspoon/lib/hotkeys.lua` 再 reload 你想要的：网页勾一下 → 记录偏好 → 下次 HS 加载时那个快捷键就不再绑定（或重新绑回）。 ## 现状澄清：Raycast vs Hammerspoon **两者完全没交集**，是不同触发模型： | 维度 | Hammerspoon | Raycast | |------|-------------|---------| | 触发 | 按键组合（全局 hotkey，按住就生效） | Raycast launcher 里打字找命令 | | 代码位置 | `~/.hammerspoon/modules/*.lua` | `~/Dev/raycast/commands/*.sh` | | 数量 | 36 快捷键 + 4 被动功能 | 42 script commands | | 是否在 dashboard | ✅ 已展示 | ❌ 完全没纳入 | ### Hammerspoon 负责（全部在 `hotkeys.lua`） - **Finder 集成**（`modules/apps.lua`）：⌘⌃⇧+T/W/I/S/R、⌘⇧+N、⌘⌥+N - **文件操作**（`modules/system.lua`）：复制文件名/内容、粘贴、压缩、打印 - **媒体控制**（`modules/media.lua`）：⌘⌃⇧+; ' L P - **系统工具**（`modules/system.lua`）：⌘⌥+,、⌃⇧+R、⌘⌃⇧+7/8/B/9 - **代理控制**（`modules/network.lua`）：⌘⌃⇧+1-6（ClashX 模式切换） - **窗口管理**（`modules/window.lua`）：⌘⌃⇧+F/M/O/Y（Yabai） - **按键重映射**（`modules/keymap.lua`）：⌘+H 删字符、⌘⌥+H 删词、⌘⌃+H 删到行首、Ctrl+HJKL vim nav、右⌥→Hyper - **被动**：右⌥Hyper、Vim 导航、微信智能启动（⌃⌥+W）、Menubar 代理指示器 ### Raycast 负责（全是 `.sh` 脚本命令，42 条） 文档/数据流水线工具，**没快捷键绑定**，靠 launcher 搜索触发： - `docx_*` / `pptx_*` / `md_*` / `csv_*` — Office 文档处理 - `hy_*` — 水利公司项目相关 - `cclog_*` — CC 会话历史 - `git_smart-push.sh`、`file_cleanup-downloads.sh` 等 **这些本来就不需要"开关"**——要么在 Raycast launcher 里用，要么不用；不占快捷键。 ## "统一化显示" —— 其实现状并不统一 你以为每个 HS 快捷键按下都有统一提示，实际 `modules/` 里混用三种： | API | 效果 | 被谁用 | |-----|------|--------| | `utils.notify(title, msg)` → `hs.notify` | 系统右上角通知中心 | apps/system/network/window 大部分 | | `utils.showInfo(msg)` → `hs.alert.show` | 屏幕中央 overlay（2s） | apps 的 run_script、showInfo | | `hs.alert.show` 直接用 | 同上 | network.show_status | | **什么都不显示** | 静默 | media（播放/暂停/切歌）、keymap 全部、apps 的 open_terminal/IDE/Nvim（部分） | **建议**：把 `utils.lua` 里统一封装成 `utils.feedback(module, action, ok)`，所有模块函数末尾调一次。这是顺手能做的收敛，不是重点任务。 ## 目标架构（推荐方案） ### 核心矛盾 - Dashboard 跑在 **VPS**，Hammerspoon 跑在 **本地 Mac** - 网页端改偏好 → 需要能"流回"本地，让 HS 下次加载时生效 ### 推荐：GitHub 作为传输层（最简、最可审计） ``` [网页端 toggle] ↓ 写入 [VPS: /opt/hs-dashboard/data/preferences.json] ↓ git push（web 侧有写权限 via PAT） [GitHub: hs-dashboard repo] ↓ 本地 pull（cron 每分钟 / git hook / fswatch） [本地: ~/Dev/hs-dashboard/data/preferences.json] ↓ symlink 或 cp [~/.hammerspoon/data/preferences.json] ↓ HS pathwatcher 触发 reload [hotkey_manager.init() 读 preferences, 被 disabled 的 key 跳过 bind] ``` **为什么不是其他方案**： - 直接 SSH 回写：VPS 得握本机 SSH key，安全面太大 - Webhook：要额外跑 daemon，复杂度高 - 本地跑 Streamlit：放弃"任何网络都能改"的初衷 ### preferences.json 结构 ```json { "version": 1, "updated_at": "2026-04-18T15:00:00Z", "hotkeys": { "apps.open_terminal": { "enabled": true }, "apps.open_ide": { "enabled": true }, "media.togglePlayback": { "enabled": false }, "keymap.delete_char": { "enabled": true } }, "passive": { "keymap.hyper": { "enabled": true }, "keymap.vim_nav": { "enabled": true }, "apps.wechat_smart_launch": { "enabled": false }, "network.menubar": { "enabled": true } } } ``` Key 用 `{module}.{func}` 作稳定 ID，改 desc 不影响。 ## 模块化实施（分阶段） ### Phase 1：只读改造 + 分类标签（不引入 VPS 回写） 产物：网页能按 Raycast/HS 分 tab 展示，HS tab 内按模块分组；每行右侧加 checkbox 但**灰色禁用**（提示"Phase 2 启用"）。 - `data/hs_config.json` 结构不动 - **新增** `data/raycast_config.json`：扫描 `~/Dev/raycast/commands/*.sh` 的 header（Raycast script 命令有 metadata 注释）生成 - `app.py` 顶层加 Tabs：`Hammerspoon` / `Raycast` / `设置` - **关键文件**： - `/Users/tianli/Dev/hs-dashboard/app.py`（改造） - `/Users/tianli/Dev/hs-dashboard/scripts/export_raycast.py`（新增，扫 Raycast 目录） ### Phase 2：preferences.json 生效链路（本地闭环） **先不上 VPS 回写，先让本地闭环跑通**。 - `app.py` checkbox 可点，直接写 `data/preferences.json` - `~/.hammerspoon/lib/preferences.lua` 新增：`load()` 返回 `{["apps.open_terminal"]=true, ...}` - `~/.hammerspoon/lib/hotkey_manager.lua` 的 `init()` 里查一下，`disabled` 的跳过 `hs.hotkey.bind` - Symlink: `~/.hammerspoon/data/preferences.json` → `~/Dev/hs-dashboard/data/preferences.json` - HS pathwatcher 已经在监 `~/.hammerspoon/**/*.lua`，需要**扩展到 `.json`**，或用独立 `hs.fs.watcher` - **关键文件**： - `/Users/tianli/.hammerspoon/lib/preferences.lua`（新增） - `/Users/tianli/.hammerspoon/lib/hotkey_manager.lua:95-131`（改 `init()`） - `/Users/tianli/.hammerspoon/init.lua:21`（扩展 pathwatcher 或加新 watcher） - `/Users/tianli/Dev/hs-dashboard/app.py`（checkbox 绑 session_state + 写 JSON） ### Phase 3：VPS 回写（GitHub 中转） - 网页写 JSON → `git commit -am` → `git push`（VPS 上 PAT） - 本地一条 `launchd` 每 60s 跑 `git pull --ff-only` 在 hs-dashboard repo - 拉到新 preferences.json → 本地 symlink 已经指向它 → HS 自动 reload ### Phase 4（可选）：通知统一化 - `utils.feedback(module, action, ok)` 替换散落的 notify/showInfo - 所有模块函数 return 前调一次，可在 preferences 里再加 `"notifications": {"compact"|"verbose"|"off"}` ## 需要你确认的决策点 以下会在 AskUserQuestion 里问： 1. **Raycast 要不要进 dashboard**：加还是不加？加的话只展示还是也要"启用/禁用"（Raycast 的启用本质是 symlink 到 `~/Library/.../extensions/`，开关语义不同，别硬套） 2. **回写路径**：先只做 Phase 1+2（本地网页也能跑，改了即生效），还是一口气做到 Phase 3（VPS + GitHub 中转）？ 3. **通知统一化**：要不要单独作为 Phase 4 做，还是现在顺手收敛在 Phase 2 里？ ## Verification（每阶段怎么验） - **Phase 1**：打开 https://hammerspoon.tianlizeng.cloud，看到 Raycast tab，里面有 42 条命令；HS tab 保持原样。 - **Phase 2**：勾掉"播放/暂停"后 reload HS，按 ⌘⌃⇧+; 无反应；重新勾上，再按有反应。`hs -c "require('lib.preferences').load().hotkeys['media.togglePlayback'].enabled"` 返回对应布尔。 - **Phase 3**：在网页改 toggle → 1 分钟内本地 HS reload，menubar/alert 能看到变化。`cd ~/Dev/hs-dashboard && git log -1` 能看到 VPS push 上来的 commit。 - **Phase 4**：每个快捷键按下都有统一格式的 alert/notify，可在 preferences.json 里改 `"notifications": "off"` 完全禁用。 ## 关键文件清单 **读/改**： -
