# 收官规划 · Playbook OS · 把工程化思维刻进全局 > 2026-04-19 · 替代原 R7 初版 plan · 本次 plan 的本质**不是站群文档**，是把用户的**工程化思维 = Playbook 方法论**作为 Claude 的元操作系统落地 --- ## 我对你工程化思维的复述（对齐颗粒度） **你要的是一套通用元协议，任何任务都按 Playbook 方式处理：** | # | 原则 | |---|---| | 1 | **任务先查 playbook，没有就立再做**（不允许先闷头动手） | | 2 | **Playbook = slash command / skill 模块化编排** — 每步一个命名动作，不是散装 bash | | 3 | **每个 domain 一个 playbook**（stations / hydro / bid / docx / web-scaffold / …），协议共享，内容独立 | | 4 | **元流程刻全局 CLAUDE.md**，所有项目走同一套入场 | | 5 | **不确定就复述 + AskUserQuestion**，颗粒度对齐才深入 | **用户决策（已确认）**： - ✅ 放 `~/Dev/configs/playbooks/` 根目录集中管理 - ✅ 入场策略：**严格 — 任何新任务先查 playbook 再动手** - ✅ 本次范围：元协议 + 站群 playbook 一套落地（作为首个实例示范） **站群只是第一个实例**。未来水利 / 标书 / DOCX / 新站搭建各自建 playbook，走同一套元协议。 --- ## Context · 为什么现在做 8 轮治理（R1-R7.2）mega menu 改了 4 次才到位，根因不是技术，是**没有成套协议**： - 每轮现场发明流程 - 教训散落在 PROGRESS / HANDOFF / session-retro / memory 多处 - 改动边界模糊，CC 常把小调整做成结构大动 用户明确：**这是最后一次；以后来一个任务 CC 的第一反应必须是"查 playbook"，不再自由发挥**。 --- ## Deliverables · 5 件产物 ### D1 · `~/Dev/configs/playbooks/META.md`（元协议 · 核心） 全局元协议文档。**任何 domain playbook 的骨架必须照它写**。 结构： - **§1 Playbook 的定义** — 什么是 playbook / 什么不是 - **§2 任何任务的入场 4 步**： ``` Step 1. /warmup（看环境 + skills + git 状态） Step 2. 查 playbooks/META.md 索引 → 匹配 domain → 读对应 playbook Step 3. 按 playbook 的编排图执行 slash command 链 Step 4. 若无对应 playbook：AskUserQuestion 拍板 → 立 playbook → 再执行 ``` - **§3 Playbook 骨架模板**（所有 domain 文件必须含）： - `§ 定位` — 这类任务是什么 / 不是什么 - `§ 入场判断` — 何时走这个 playbook（trigger words / 反例） - `§ 编排图` — ASCII slash command 串联 - `§ 复用清单` — 现有 skill / command / MCP - `§ 改动边界矩阵` — 破坏 vs 非破坏，对应不同入口 - `§ 踩坑 anti-patterns` — 本 domain 专属 - `§ 文件索引` — 关键文件路径 - `§ 扩展机制` — 新增子能力要动哪些点 - **§4 全局硬规则**： - 不确定 → AskUserQuestion 复述 + 问 - 颗粒度对齐才深入 - 破坏性操作先批准 - 手工 bash 做本该有 skill 的事 = 错，用户可纠正「用 /xxx」 - 复盘默认 playbook 风格（见 `/retro`），不写工具调用清单 - **§5 当前 domains 索引表**： | domain | playbook 文件 | 状态 | |---|---|---| | stations | stations.md | ✅ 已立（本次） | | hydro | hydro.md | 🚧 未立 | | bid | bid.md | 🚧 未立 | | docx | docx.md | 🚧 未立 | | web-scaffold | web-scaffold.md | 🚧 未立 | ### D2 · `~/Dev/configs/playbooks/stations.md`（站群 playbook · 首个实例） 按 META §3 骨架写。内容对 R1-R7.2 所有经验沉淀： - **§ 定位**：tianlizeng.cloud 26+ 子域 / 6 活跃站 / SSOT yaml → render → audit → sync → deploy 5 步协议 - **§ 入场判断**：触发词 "改 navbar / 改菜单 / 改样式 / 加子域 / 部署站群 / ..."；反例 "改业务代码 / 改 hydro 计算 ..." - **§ 编排图**：两条路径 - 前端微调（90%）：`/warmup` → `/frontend-tweak` → `/site-refresh-all` → `/deploy × N` → `/ship` -
