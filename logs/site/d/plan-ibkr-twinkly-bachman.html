# 站群治理 Skill 补全 · web-scaffold playbook 立项 · slash 生态整顿 > 计划日期：2026-04-19 > 触发：assets.tianlizeng.cloud 上线时，我手工做了 `/ship-site` 本就覆盖的事（ssh 写 nginx、/cf 登记、首次 deploy），是 stations.md anti-pattern #7 的严重违反。 > 前一个 plan（换汇研究）已完成，此文件被覆写。 --- ## Context — 为什么 1. **发现 gap**：本次 assets 站从零上线时，我没调 `/ship-site` / `/launch`，手工 ssh VPS 写 nginx + 手工跑 `/cf origin add` + 手工 `bash deploy.sh`。读源文件后发现 `/ship-site` 已经完整覆盖这些动作 —— 是**漏用 skill**，不是"缺 skill"。 2. **根因**：`/site-add` 没有支持 md-frontmatter→html 这类 template（目前只有 stack / changelog / docs 三种），我就"自然"跳过 `/site-add`，跳过后连带跳过了 `/ship-site` 的心智默认。 3. **meta 层面**：52 个 slash command 存在局部冗余（/recap vs /handoff 的 recap 阶段、/retro vs /distill 的 playbook 产出），职责边界需要在文档层面收敛，否则下一个会话还会"漏用/重复用/用错"。 4. **playbook OS 空缺**：`~/Dev/configs/playbooks/META.md §5` 里 `web-scaffold` 状态 🚧 未立，本次恰好是首次实战该 domain，正好沉淀。 **预期结果**：下次开新静态站（任意 template），CC 能准确走 `/launch` 或 `/site-add + /ship-site` 链路，不再手工；slash 生态有自检机制；本次漏用的 skill 有 playbook + retro 双重记录以供未来纠错。 --- ## Deliverables（五件套） ### D1 · `/site-add` 扩 md-docs template（代码修改） - 文件：`~/Dev/cc-configs/commands/site-add.md` - 改动：`--template` 参数从 `stack|changelog|docs` → `stack|changelog|docs|md-docs` - 新增第 7 小节「md-docs 模板说明」： - 源：跨仓读 `~/Dev/<source-repo>/docs/*.md` 过滤 frontmatter `public: true` -
