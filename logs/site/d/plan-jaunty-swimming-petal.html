Auggie Dashboard 大幅简化

PLAN · (plan) · 2026-04-20 20:49 · ~/.claude/plans/jaunty-swimming-petal.md
# Auggie Dashboard 大幅简化 ## Context 用户要的就 3 件事： 1. **一个域名实时看**"我有几个 GitHub repo，auggie 能搜几个" 2. **新推的 GitHub repo 自动出现**在这个清单里 3. **只走 MCP，不碰 CLI**（已落规则） 现状拆解后发现过度复杂： - `scan.json` 一条有 **13 个字段**（`local_ready`/`cloud_ready`/`cloud_complete`/`has_claude_dir`/`has_files_md`/`binary_count`/`category` 等）—— 大部分字段是在回答"本地+云端哪边 ready"这种已失时义的问题 - `labs/auggie-dashboard/` 里 **app.py + deploy.sh + requirements.txt** 是旧 Streamlit 站遗产，旧站已归档，这些是死代码 - `fix_files_md.py` 122 行在维护 `_files.md` 二进制清单机制 —— auggie 走 MCP 实时检索以后不需要这个 - Scanner 走**本地文件系统扫描**（扫 `~/Dev` + `~/Work` 找 `.git` 目录）—— 和"GitHub 上有几个 repo"不是一个问题，而且本地没 clone 的 repo 根本看不到 ## 目标 `dashboard.tianlizeng.cloud/auggie` 就是一张表：**GitHub 上 zengtianli 账号的全部 repo**（当前 61 个：26 公开 + 35 私有），name / visibility / language / last push / description。新推 repo → 下次刷新自动出现。 ## 执行步骤 ### Step 1 — Scanner 重写（180 行 → 约 30 行） 改 `~/Dev/labs/auggie-dashboard/lib/scanner.py` 逻辑： - 丢弃本地文件系统扫描 - 只调一次 `gh api user/repos --paginate`（Mac 本地跑，gh 已 auth） - 输出**7 字段 schema**：`[{name, visibility, language, pushed_at, description, html_url, archived}]` - 61 个 repo，JSON 预估 ~8KB ### Step 2 — 删除 Streamlit 遗产 ``` labs/auggie-dashboard/ ├── app.py ← 删（Streamlit UI，已替换为 ops-console Next.js） ├── deploy.sh ← 删（部署 Streamlit 到 VPS，已归档） ├── requirements.txt ← 删（Streamlit 依赖） ├── uv.lock ← 删（regenerate） ├── lib/ │ ├── fix_files_md.py ← 删（_files.md 机制已废） │ └── scanner.py ← 重写（见 Step 1） ├── data/ │ └── scan.json ← 用新 schema 重建 └── pyproject.toml ← 瘦身依赖，只留 scanner 需要的 ``` ### Step 3 — ops-console `/auggie` 页面简化 改 `~/Dev/stations/ops-console/app/auggie/page.tsx`： - 指标卡：**总数** · **公开** · **私有** · **archived** - 单表格：Name / Visibility / Language / Last Push / Description / GitHub 链接 - 客户端搜索框（按 name 过滤） - `revalidate: 60` 让 Next.js 每 60s 重拉一次 `data/auggie-scan.json` 改 `app/api/auggie/route.ts`：schema 同步（drop `status` / `cloud_ready` 等旧字段） ### Step 4 — `/auggie dash` 手动刷新链路 用户按需跑 `/auggie dash` 时，做这 3 件事： 1. `python3 ~/Dev/labs/auggie-dashboard/lib/scanner.py data/scan.json` — 拉 gh api → 写 scan.json 2. `cp data/scan.json ~/Dev/stations/ops-console/data/auggie-scan.json` — 同步到消费者 3. `scp` 到 VPS `/opt/ops-console/data/auggie-scan.json` — 免 Next.js rebuild 改 `~/Dev/tools/cc-configs/commands/auggie.md`： - **移除 `fix` 子命令**（`_files.md` 机制已废） - `dash` 子命令 = 上面这 3 步 - 默认子命令改成 `dash` ## 改动文件清单 | 文件 | 动作 | |---|---| | `labs/auggie-dashboard/lib/scanner.py` | 重写（40 行以内） | | `labs/auggie-dashboard/data/scan.json` | 重建（新 schema） | | `labs/auggie-dashboard/pyproject.toml` | 瘦身 | | `labs/auggie-dashboard/app.py` | **删** | | `labs/auggie-dashboard/deploy.sh` | **删** | | `labs/auggie-dashboard/requirements.txt` | **删** | | `labs/auggie-dashboard/uv.lock` | **删** | | `labs/auggie-dashboard/lib/fix_files_md.py` | **删** | | `stations/ops-console/app/auggie/page.tsx` | 简化渲染 | | `stations/ops-console/app/api/auggie/route.ts` | schema 同步 | | `stations/ops-console/data/auggie-scan.json` | 用新 schema 重建 | | `tools/cc-configs/commands/auggie.md` | 移除 `fix` 子命令、`dash` 改为 3 步流程 | ## 不动的东西 - **Auggie MCP 配置** (`~/.claude.json` + `~/.augment/session.json` + `~/.claude/CLAUDE.md` 规则) — 上轮已搞定，这轮不碰 - **auggie.tianlizeng.cloud 子域** — 已归档，不重建（用 `dashboard.tianlizeng.cloud/auggie`） - **VPS systemd / nginx** — `/auggie` 是 ops-console 内部路由，不需要独立服务 ## 验证 1. 本地：`python3 ~/Dev/labs/auggie-dashboard/lib/scanner.py data/scan.json` → `jq length` = 61 2. `cat data/scan.json | jq '.[0]|keys'` 只应看到 7 个字段 3. 部署后：`curl https://dashboard.tianlizeng.cloud/api/auggie | jq 'length'` = 61 4. `curl .../api/auggie | jq '.[0]|keys'` 同样只 7 字段 5. 浏览器打开 dashboard.tianlizeng.cloud/auggie → 表格 61 行 + 4 张指标卡 6. 手动 push 新 repo → 跑 `/auggie dash` → 刷新页面能看到新行 ## 风险 / 代价 - 旧 `scan.json` 里的 `status`/`category`/`cloud_ready`/`has_files_md` 等字段会**消失**，ops-console 页面需同步改 - 手动刷新 = 新 repo 不会自动出现。用户要每次推完跑 `/auggie dash`（或以后再加 cron）
