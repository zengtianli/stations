# 应用 6 子域治理命令：修 auggie + 归档 hammerspoon + 真跑 rename cc ## Context 上一轮（2026-04-17 上半场）已完成 6 个 slash 命令的建造 + Streamlit navbar 修复。本轮把工具真跑在生产 infra 上： 1. **auggie 400 的根因已定位**：`/cf-audit` 报告 `auggie.tianlizeng.cloud` 在 services.ts 声明了 port 8522 但没加到任何 Origin Rule 的 expression 里，CF 边缘不知道 routing 到 8522，才 400。一行命令可修。 2. **hammerspoon 是 CF DNS + VPS nginx 双孤儿**：services.ts 里没它，DNS 和 nginx config 却还在。`/site-archive` 正是为此设计。 3. **`cc → cc-options` 这个改名是 2026-04-17 handoff 写进去的目标**（"cc 代表 covered call，消除歧义"）。工具 dry-run 通过，真跑是首次使用 `/site-rename` 命中生产。 本轮目标：用刚建好的工具跑完上述 3 个动作，验证工具，清理 `/cf-audit` 报告里最具体的 3 个漂移。刻意不做 hydro-\* 归档（需要先开 log_format + 等一周真实 7d 数据）和 nginx log_format 改造（下一轮独立做）。 ## 当前状态（刚确认） ``` | auggie | ❌ 400 | origin rule missing | | cc | 🔒 302 | healthy, port 8521, cf-access | | hammerspoon DNS | 🟠 A | orphan in CF, no services.ts | | services.ts | clean | OK for /site-rename preflight | | projects.yaml | clean | OK for /site-rename preflight | ``` --- ## 步骤 ### 1. 修 auggie 400（单行，0 风险） ```bash source ~/.personal_env python3 ~/Dev/devtools/lib/tools/cf_api.py origin-rules add auggie 8522 ``` - **修什么**：把 `"auggie.tianlizeng.cloud"` 加到现有 port=8522 的 Origin Rule expression。cf_api.py 里 `origin_rules_add()` 已实现，查找匹配 port 的 rule → PATCH expression。 - **验证**：`source ~/.personal_env && python3 ~/Dev/devtools/lib/tools/sites_health.py --filter auggie` → HTTP 应从 ❌ 400 变成 ✅ 200 或 🔒 302（需看 services.ts 是否声明了 cf-access，auggie 当前声明 public 应是 200） - **回滚**：`cf_api.py origin-rules remove auggie` ### 2. 归档 hammerspoon 孤儿（中风险，可恢复） ```bash source ~/.personal_env python3 ~/Dev/devtools/lib/tools/site_archive.py hammerspoon --yes ``` - **site_archive.py 会做什么**（hammerspoon 特定情境）： - nginx_enabled=True → `rm /etc/nginx/sites-enabled/hammerspoon{,.tianlizeng.cloud}` + reload（sites-available 保留） - systemd unit=None → 跳过 - CF DNS delete `hammerspoon.tianlizeng.cloud` - services.ts 里本来没它 → "no entry" warning（正常） - projects.yaml 里有条 `- name: hammerspoon` → `status: archived` - /var/www/hammerspoon 不存在 → 跳过 mv - **验证**： - `curl -sI https://hammerspoon.tianlizeng.cloud` 应为 NXDOMAIN 或 CF 404 - `python3 cf_audit.py` 的 `orphan-in-cf-dns` 和 `nginx-orphan` 这两项应变 0 - **回滚**： - `cf_api.py dns add hammerspoon` - VPS：`ln -sfn /etc/nginx/sites-available/hammerspoon{,.tianlizeng.cloud} /etc/nginx/sites-enabled/` + reload - `git -C ~/Dev/stack checkout -- projects.yaml` ### 3. 真跑 `/site-rename cc cc-options`（高风险，首次实盘） ```bash source ~/.personal_env python3 ~/Dev/devtools/lib/tools/site_rename.py cc cc-options --yes ``` - **动作链**（已 dry-run 通过）： 1. Pre-flight：services.ts + projects.yaml git clean、cc 存在、cc-options 不存在（所有验证在 site_rename.py `preflight()`） 2. `cf_api.dns_add("cc-options")` → A 记录 → VPS_IP，proxied 3. `cf_api.origin_rules_add("cc-options", 8521)` → 现有 port=8521 rule expression 加 host 4. `cf_api.access_add("cc-options")` → 沿用 `zengtianli1@gmail.com` 5. VPS（一条 SSH）： ``` cp /etc/nginx/sites-available/cc.tianlizeng.cloud → cc-options.tianlizeng.cloud sed 's/cc.tianlizeng.cloud/cc-options.tianlizeng.cloud/g' 新文件 ln -sfn ...sites-available/cc-options.tianlizeng.cloud sites-enabled/ cp cc.tianlizeng.cloud → cc.tianlizeng.cloud.prerename.bak printf 新 301 vhost > cc.tianlizeng.cloud nginx -t && systemctl reload nginx ``` 6. `_update_services_ts`：精确 regex 只改那条 entry 的 `subdomain: "cc"` → `"cc-options"` 7. `_update_projects_yaml`：文本替换 `cc.tianlizeng.cloud` → `cc-options.tianlizeng.cloud` 8. `_schedule_archive` 写 `~/Dev/configs/scheduled-archives.json`：2026-05-01 归档老 cc - **验证**（按此顺序）： 1. `curl -sI https://cc-options.tianlizeng.cloud` → 应 302 到 `cloudflareaccess.com` 2. `curl -sI https://cc.tianlizeng.cloud` → 应 301 到 `https://cc-options.tianlizeng.cloud/` 3. `source ~/.personal_env && python3 sites_health.py --filter cc` → 两行都应健康 4. `python3 cf_audit.py` → 应无新漂移 5. `cat ~/Dev/configs/scheduled-archives.json` → 有 cc 条目，scheduled_archive = 2026-05-01 - **真正的风险点**： - nginx sed 对 `cc.tianlizeng.cloud` 字符串做全局替换。老配置若包含 regex 意义的字符（例如 location 块里提到 `cc.tianlizeng.cloud`），可能漏改或错改。**缓解**：sed 跑完后 `nginx -t` 会在 reload 前验语法，失败则老流量不中断（cc 老 vhost 已备份到 `.prerename.bak`） - CF DNS proxied=True + Origin Rule patch 之间有约 30s 边缘传播。期间 cc-options 可能 522/500。等 60s 再 curl - **回滚**（按顺序一步步做）： 1. `cf_api.py access delete cc-options` 2. `cf_api.py dns delete cc-options` 3. `cf_api.py origin-rules remove cc-options` 4. SSH： ``` rm /etc/nginx/sites-{enabled,available}/cc-options.tianlizeng.cloud cp /etc/nginx/sites-available/cc.tianlizeng.cloud.prerename.bak ...cc.tianlizeng.cloud nginx -t && systemctl reload nginx ``` 5. `git -C ~/Dev/website checkout -- lib/services.ts` 6. `git -C ~/Dev/stack checkout -- projects.yaml` 7. `rm ~/Dev/configs/scheduled-archives.json`（如只有这条）或编辑删 cc 项 ### 4. 收尾提交 + 推送 ```bash # services.ts & projects.yaml 已被 Step 2/3 修改 cd ~/Dev/website && git add lib/services.ts && \ git -c commit.gpgsign=false commit -m "chore(services): cc → cc-options + archive hammerspoon" git push cd ~/Dev/stack && git add projects.yaml && \ git -c commit.gpgsign=false commit -m "chore(projects): mark hammerspoon archived + cc renamed" git push ``` - website 的 GHA 会自动重建首页，cmds/stack 站的 navbar 链接对 cc 的引用**应该没有**（navbar 里的链接只涉及 Home/Stack/CC Docs/Changelog/Dashboard，没有 cc.tianlizeng.cloud）—— 这个 Step 4 开始前用 `grep -r "cc.tianlizeng.cloud" ~/Dev/devtools/lib/templates/ ~/Dev/website/` 验证一次。 - `~/Dev/configs/scheduled-archives.json` 不用 commit（configs 是 untracked 工作数据） ### 5. 最终验证 + 更新 HANDOFF.md ```bash source ~/.personal_env python3 ~/Dev/devtools/lib/tools/sites_health.py 2>&1 | tee /tmp/sites-health-post.md python3 ~/Dev/devtools/lib/tools/cf_audit.py ``` 期望： - `/sites-health`：auggie 从 400 变好；cc 和 cc-options 都 302（Access）；hammerspoon 不在列表 - `/cf-audit`：drift 数从 6 降到 ≤2（只剩 changelog / dashboard / tianlizeng.cloud 这种架构性 drift，无 bug） 最后：edit `/Users/tianli/Dev/HANDOFF.md` 记录本轮真跑结果 + 下轮待办（hydro 批量归档、nginx log_format 改造）。 --- ## 本轮明确不做 | 任务 | 理由 | 下轮处理 | |---|---|---| | 6 个 hydro-\* 归档 | 没有真 7d 数据，handoff 里说"6 个"但不知具体哪 6 个 | 开了 nginx log_format 等一周后基于数据判断 | | nginx `$host` log_format 改造 | 改 VPS 全局配置属独立变更，不应和应用层操作混一轮 | 下一轮独立做 + 验证 + 等数据 | | 浏览器目测 Streamlit navbar | 我无浏览器能力 | 用户做 | | 命名建议表里其他改名（panel/sub/docs/n8n） | 用户没说现在改；本轮先验 `/site-rename` 在 cc 上稳，下轮再批量 | 下轮 | --- ## 关键文件（本轮涉及） -
