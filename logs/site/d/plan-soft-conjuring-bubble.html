# CC API 管理平台（仿 MMKG CODE） ## Context 用户有 4 个 CC Pro 订阅，已有一个能跑通的 OAuth proxy（`oauth_proxy.py`），现在要把它升级为一个**像 MMKG CODE 一样的完整 API 管理网站**，部署在 `proxy.tianlizeng.cloud`。 现有资源： - VPS `104.218.100.67`：4C/8G，Node 22，Python 3.10，Docker，Nginx，51G 空间 - 域名 `tianlizeng.cloud`，Cloudflare SSL（origin cert 已有） - 已跑通的 proxy 后端（`oauth_proxy.py`，aiohttp，token 轮换/刷新/sticky routing） - Private repo: `zengtianli/oauth-proxy` 参考：MMKG CODE (`code.mmkg.cloud`) — SPA 前端 + REST API 后端 ## 架构设计 ``` proxy.tianlizeng.cloud ├── / → 前端 SPA (React/Vite) ├── /api/... → 管理 API (Node/Express 或 Python) ├── /v1/messages → Claude API 代理 (已有 oauth_proxy.py) └── /v1/messages/count_tokens ``` ### 技术选型 - **前端**: React + Vite + TailwindCSS（单页应用，参考 MMKG 的侧边栏布局） - **后端**: 复用现有 `oauth_proxy.py`（aiohttp），扩展管理 API - **数据库**: SQLite（单用户，足够了，不需要 PostgreSQL） - **部署**: 直接跑，Nginx 反代，systemd ### 为什么不用 Node 重写 现有 proxy 已经跑通了（token 刷新、轮换、SSE 流式），没必要重写。只需要在 aiohttp 上加管理 API + 静态文件服务。 ## 核心功能（只做你需要的） ### 1. 登录页 - 简单的用户名+密码登录（单用户，config.json 里配置） - JWT token 保持会话 - 不需要注册、邮箱验证等 ### 2. 仪表盘 Dashboard (`/`) - 今日/近 30 天请求数、token 用量 - 4 个后端账号状态卡片（healthy/error/skip） - API 端点展示：`https://proxy.tianlizeng.cloud` ### 3. API 密钥管理 (`/keys`) - 创建/删除/启用/禁用 API key - 每个 key 有名称、分组、用量统计、状态 - 表格展示（和 MMKG 截图一模一样的布局） - 复制 key 按钮 ### 4. 使用记录 (`/usage`) - 最近 N 条 API 调用记录：时间、模型、input/output tokens、用了哪个后端账号、状态码 - 按日汇总图表 ### 5. 状态面板 (`/status`) - 4 个后端账号详情：名称、email、状态、token TTL、refresh 失败次数 - 启用/禁用开关（替代现有 toggle） ### 不做的功能 - 充值/订阅、兑换（自用不需要） - 工单中心、在线客服 - 模型定价 - 多用户注册 ## 实现步骤 ### Step 1: 后端扩展 `oauth_proxy.py` **文件**: `~/Dev/oauth-proxy/oauth_proxy.py` 新增： - **SQLite 数据库**: `proxy.db` - `api_keys` 表: id, name, key, group_name, enabled, created_at - `usage_logs` 表: id, api_key_id, timestamp, model, input_tokens, output_tokens, cache_read_tokens, cache_creation_tokens, account_name, status_code, latency_ms - **JWT 登录**: `POST /api/auth/login` → 返回 JWT；config.json 加 `admin_user` / `admin_password_hash` - **API Key CRUD**: - `GET /api/keys` → 列表（含用量统计） - `POST /api/keys` → 创建（自动生成 `sk-proxy-xxx`） - `DELETE /api/keys/{id}` → 删除 - `PATCH /api/keys/{id}` → 编辑名称/分组/启用状态 - **使用记录**: - `GET /api/usage?days=30` → 使用日志 - `GET /api/usage/summary` → 汇总统计 - **账号状态**: `GET /api/accounts` （替代 /health） - **账号开关**: `POST /api/accounts/{name}/toggle` - **修改 `handle_proxy`**: - 验证改为查 `api_keys` 表（而不是单一 proxy_api_key） - 每次请求记录到 `usage_logs`（从上游响应提取 token 数） - **静态文件服务**: `GET /` → 返回前端 `dist/` 目录 ### Step 2: 前端 SPA **目录**: `~/Dev/oauth-proxy/frontend/` ``` frontend/ ├── package.json ├── vite.config.js ├── tailwind.config.js ├── index.html └── src/ ├── main.jsx ├── App.jsx ├── api.js (fetch wrapper, JWT header) ├── pages/ │ ├── Login.jsx │ ├── Dashboard.jsx │ ├── Keys.jsx │ ├── Usage.jsx │ └── Status.jsx └── components/ ├── Layout.jsx (侧边栏 + topbar) ├── Sidebar.jsx └── KeyTable.jsx ``` 布局仿 MMKG： - 左侧深色侧边栏（仪表盘/API密钥/使用记录/状态面板） - 右侧内容区 - 顶栏显示用户名 + API 端点 ### Step 3: 构建 & 部署 1. 本地 `cd frontend && npm run build` → 生成 `dist/` 2. `deploy.sh` 更新为同步 `dist/` + `oauth_proxy.py` + `proxy.db`（如首次）到 VPS 3. Nginx 配置不变（proxy_pass 到 9100，aiohttp 既服务 API 又服务静态文件） ### Step 4: 初始化 - 首次启动时自动创建 SQLite 表 - config.json 新增 `admin_user`、`admin_password_hash`、`jwt_secret` - 自动迁移：现有 `proxy_api_key` 导入为第一个 API key ## 数据库 Schema ```sql CREATE TABLE api_keys ( id INTEGER PRIMARY KEY AUTOINCREMENT, name TEXT NOT NULL, key TEXT UNIQUE NOT NULL, group_name TEXT DEFAULT '', enabled INTEGER DEFAULT 1, created_at TEXT DEFAULT (datetime('now')) ); CREATE TABLE usage_logs ( id INTEGER PRIMARY KEY AUTOINCREMENT, api_key_id INTEGER, timestamp TEXT DEFAULT (datetime('now')), model TEXT, input_tokens INTEGER DEFAULT 0, output_tokens INTEGER DEFAULT 0, cache_read_tokens INTEGER DEFAULT 0, cache_creation_tokens INTEGER DEFAULT 0, account_name TEXT, status_code INTEGER, latency_ms INTEGER DEFAULT 0, FOREIGN KEY (api_key_id) REFERENCES api_keys(id) ); ``` ## 关键文件 - `~/Dev/oauth-proxy/oauth_proxy.py` — 后端（扩展） - `~/Dev/oauth-proxy/frontend/` — 前端 SPA（新建） - `~/Dev/oauth-proxy/config.json` — 配置（新增 admin_user 等） - `~/Dev/oauth-proxy/deploy.sh` — 部署脚本（更新） ## 验证 1. 打开 `https://proxy.tianlizeng.cloud` → 跳转登录页 2. 登录后看到仪表盘 3. 在 API 密钥页创建一个 key 4. 用新 key curl `/v1/messages` → 成功 5. 使用记录页看到刚才的请求 6. 状态面板看到 4 个账号状态
