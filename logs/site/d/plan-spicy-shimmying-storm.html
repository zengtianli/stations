# 不要重复造轮子 — 教训总结与清理 ## Context 本轮对话犯了一个严重错误：用户要求邮箱验证码登录时，我自己写了一整套 SMTP 发送验证码的后端逻辑（aiosmtplib + 126.com SMTP + 验证码存储 + 过期机制 + 前端两步流程），**完全无视了 docs.tianlizeng.cloud 已经在用 Cloudflare Access 做同样的事情**。 CF Access 是现成的、已部署的、用户熟悉的认证模块。我却花了大量时间造了一个劣质替代品。 根因：**没有先调查现有方案就开始写代码**。 --- ## 已完成的工作（上一轮） 以下已完成且正确，不需要改动： - ✅ HTTPS: proxy.tianlizeng.cloud (CF Proxied + Origin Rule 8443) - ✅ CF Access: proxy.tianlizeng.cloud 已配置（Allow zengtianli1@gmail.com + /v1 bypass） - ✅ CLAUDE.md: oauth-proxy/CLAUDE.md 和 ~/Dev/CLAUDE.md 已创建/更新 - ✅ deploy.sh: 已升级含前端构建+同步 - ✅ ~/.claude/commands/: deploy, vps-status, cf-dns 三个命令 - ✅ scripts/lib/tools/: cf_api.py + vps_cmd.py 已创建并推送 GitHub - ✅ auggie: 可以搜到所有新文件 - ✅ GitHub: oauth-proxy (978decf) + scripts (431895e) 都已推送 --- ## 还需要做的清理工作 ### 1. 清理 oauth_proxy.py 中的死代码 CF Access 已经接管了 dashboard 认证，以下代码变成死代码，应删除： **删除：** - `import aiosmtplib` - `from email.mime.text import MIMEText` - `import random`（仅用于验证码生成） - `handle_send_code()` 整个函数 - `handle_login()` 中的邮箱验证码逻辑 - `app["verify_codes"]` 初始化 - `app.router.add_post("/api/auth/send-code", ...)` 路由 - `app.router.add_post("/api/auth/login", ...)` 路由 - `ensure_admin_config()` 中的 SMTP 相关字段（smtp_host/smtp_port/smtp_user/smtp_password/admin_email） - JWT 相关代码（`create_jwt`, `verify_jwt`, `require_auth` 装饰器）— CF Access 在边缘已认证 - `@require_auth` 装饰器从所有 admin API handler 上移除 **保留：** - `hashlib`（如果其他地方用到） - API key 验证逻辑（`validate_api_key` — 这是 proxy API 的认证，和 dashboard 无关） **结果：** 后端不再有自定义认证，dashboard 完全依赖 CF Access，API 路径用 x-api-key 认证。 ### 2. 清理前端 Login.jsx CF Access 拦截发生在 Cloudflare 边缘，通过 CF 认证后请求才到达后端。前端不需要自己的登录页。 **删除/简化：** - `frontend/src/pages/Login.jsx` — 删除整个文件 - `frontend/src/App.jsx` — 移除 PrivateRoute 和 /login 路由，所有页面直接显示 - `frontend/src/api.js` — 移除 JWT token 相关代码（getToken/setToken/clearToken） ### 3. 更新 CLAUDE.md 和记忆 **`/Users/tianli/Dev/oauth-proxy/CLAUDE.md`：** - 认证改为 "CF Access（和 docs.tianlizeng.cloud 同模式）" - 删除 SMTP 相关描述 **`/Users/tianli/Dev/CLAUDE.md`：** - 用户偏好改为 "登录用 CF Access（Cloudflare Zero Trust 邮箱验证码），不要自建认证" **新增记忆 `feedback_no_reinventing.md`：** 核心教训： 1. **做任何功能前，先查现有项目是否已有同类方案** 2. 用户的其他站点（docs/status/n8n）已经用了 CF Access → 新站点也用 CF Access 3. 认证/DNS/SSL/部署 这些基础设施层面的东西，优先用 Cloudflare/nginx 等已有平台功能 4. 只在平台功能覆盖不到的地方写自定义代码 ### 4. cf_api.py 扩展 — 加 CF Access 管理 既然 CF Access 是标准认证模块，cf_api.py 应该能管理它： -
