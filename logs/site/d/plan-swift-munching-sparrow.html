# 改进计划：基于 /insights 报告 **来源**：/insights 报告分析 + 本次会话讨论 **审查状态**：待用户拍板，按编号独立可选 --- ## 已完成（本次会话） | 已做 | 文件 | |---|---| | CLAUDE.md 加 Execution Discipline / Diagnostic Approach / Hook & Config Safety 三节 | `~/.claude/CLAUDE.md` | | 新建 `/preflight` 命令 | `~/.claude/commands/preflight.md` | | `/deploy` 加 Pre-flight Step 0 + 真实验证 | `~/.claude/commands/deploy.md` | | `/ship-site` 加 Step 0 并行预检 + 暂停确认 | `~/.claude/commands/ship-site.md` | | `/deploy-changed` 加 Pre-flight 变更清单 + 确认 | `~/.claude/commands/deploy-changed.md` | --- ## 待审查的改进项 --- ### 改进 A：PreToolUse Hook — 拦截危险删除 **是什么** 在 `~/.claude/settings.json` 里加一条 `PreToolUse` hook，在 Claude 执行任何 Bash 命令前，检查命令文本是否包含危险删除模式，匹配则阻断并输出提示，要求用户手动确认。 **为什么加** - 报告记录的真实损失：`site-content.css` 被当作重复文件删除（实为部署产物）；Stop hook 被删；`HANDOFF.md` 被意外覆盖 - 当前 `settings.json` 的 `deny` 规则只能拦截精确命令（如 `rm -rf ~/`），对 `rm *.css`、`rm -r ./site-content.css` 这类模式无效 - PreToolUse hook 在命令执行前运行，可以用 grep 做模式检查，阻断力更强 **拦截的模式** ``` rm .*\.css → CSS 文件（部署产物风险） rm .*\.md → Markdown 文件（HANDOFF/CLAUDE.md 风险） rm .*HANDOFF → 交接文件 rm .*symlink|unlink → 符号链接 pkill -f.*ssh → 可能杀掉自己的 SSH session（历史事故） ``` **使用场景** - Claude 在整理目录时把 `site-content.css` 的副本当重复文件删除 → 被拦截，Claude 必须先解释为什么安全 - Claude 用 `pkill -f python` 或 `pkill -f ssh` 时 → 被拦截提示确认 - Claude 删 `HANDOFF.md` 时 → 被拦截 **不拦截的** - `/tidy` skill 内的清理（已经有审批流程） - 用户自己在终端运行的命令（hook 只影响 Claude 发出的 Bash 调用） **改动范围** `~/.claude/settings.json` — 在 `hooks` 节加 `PreToolUse` 数组 ```json "PreToolUse": [ { "matcher": "Bash", "hooks": [ { "type": "command", "command": "cmd=$(echo \"$CLAUDE_TOOL_INPUT\" | python3 -c \"import sys,json; d=json.load(sys.stdin); print(d.get('command',''))\"); echo \"$cmd\" | grep -qE 'rm[[:space:]].*\\.css|rm[[:space:]].*\\.md|rm[[:space:]].*HANDOFF|pkill.*ssh' && echo 'BLOCKED: 危险删除操作 — 请先说明该文件非部署产物，再由用户确认执行' && exit 2 || exit 0", "timeout": 3 } ] } ] ``` **风险** - 低：仅拦截特定模式，不影响正常操作 - 误拦截率：偶尔会拦截合法的 `.md` 文件删除（如清理临时文档），但 Claude 说明理由后用户确认即可 - 如有误伤可随时调整 grep 表达式 --- ### 改进 B：CLAUDE.md 补 HANDOFF 放置规则 **是什么** 在现有 CLAUDE.md 的「工作字典」硬规则节，加一行： `HANDOFF.md 放到改动最多的项目目录下，不放全局根 ~/Dev/HANDOFF.md` **为什么加** - 报告指出用户在多个会话里纠正过 HANDOFF.md 的放置位置 - `/handoff` 命令本身有启发式目录检测，但没有在 CLAUDE.md 里写明规则，所以偶尔还会放错 **使用场景** - 做完站群改动，HANDOFF 自动放到 `~/Dev/stations/web-stack/HANDOFF.md` 而非全局根 **改动范围** `~/.claude/CLAUDE.md` — 在「硬规则」列表加 1 行 --- ### 改进 C：新命令 `/fanout-deploy` — 并行多站部署编排 **是什么** 新建 `~/.claude/commands/fanout-deploy.md`。接收一个变更规格（如「把所有站的 navbar 标题改成 X」）和目标 repo 列表，为每个站点并行起一个 Agent subagent，各自完成 edit + build + deploy + verify，最终汇总成一张状态表。 **为什么加** - 报告数据：你有 27 个子域、13 个 repo，每次跨站改动都是串行跑，工具使用（Bash 2980 次）大量时间浪费在等待 - 现有 `/deploy-changed` 只能自动检测改了哪些站，但底层 `deploy-batch.sh` 仍是串行（或有限并行） - **报告原话**：「你的工作时间能在站点并行部署时大幅压缩」 **和现有命令的区别** | 命令 | 适用场景 | |---|---| | `/deploy` | 当前目录单站，有 deploy.sh | | `/deploy-changed` | 从 git diff 自动发现受影响站，批量部署 | | `/ship-site` | 新站首次上线（含 CF DNS/Access 创建） | | **/fanout-deploy（新）** | 已知要改哪些站，并行 edit+deploy+verify，适合跨站 SSOT 同步 | **使用场景** ``` /fanout-deploy "把所有站的 navbar 版本号从 2.1 改到 2.2" \ --repos stations/web-stack,hydro-annual,hydro-capacity ``` - 场景 1：navbar 模板改了，需要同步到 6 个消费者站，每个站各自重新 build + deploy + verify，全部并行，最终报告哪些成功哪些失败 - 场景 2：menus.yaml 改完，需要推送所有静态站，不想串行等 10 分钟 **输出格式** ``` ## Fanout Deploy Results | Repo | Status | Commit SHA | Live URL | HTTP | |-------------------|---------|------------|-----------------------------|------| | hydro-annual | ✅ done | abc1234 | hydro-annual.tianlizeng.cloud | 302 | | hydro-capacity | ❌ fail | — | — | err | | stations/web-stack| ✅ done | def5678 | stack.tianlizeng.cloud | 302 | ``` **改动范围** 新建 `~/.claude/commands/fanout-deploy.md`（约 60 行） **技术实现** - 用 Claude 的 `Agent` tool 为每个 repo 起独立 subagent - 每个 subagent 接收：repo 路径 + 变更规格 + 验证 URL，返回结构化 JSON - 编排层汇总所有返回，生成 markdown 表格 - 支持 `--dry-run`（只打印变更 diff，不执行） - 支持 `--repos` 指定目标或 `--all`（从 services.ts 读全量） --- ### 改进 D：新命令 `/autonomous-refactor` — 自循环重构 **是什么** 新建 `~/.claude/commands/autonomous-refactor.md`。接收：重构目标 + 验收脚本路径 + 最大迭代次数。Claude 在 git worktree 里自循环：改代码 → 运行验收脚本 → 解析 failures → 修 top failure → 重复，直到全通过或耗尽预算。仅在完成或耗尽时通知用户。 **为什么加** - 报告数据：你在单会话跑了 27 commits、每次都要等你审批、Claude 返回 buggy 代码 25 次 - 报告原话：「定义验收测试，让 Claude 自循环 patch-test-fix，变成过夜任务」 - **适合场景**：大型重构（如迁移到 Turborepo）可以在夜间自跑，只在完成时叫你 **使用场景** ``` /autonomous-refactor \ --goal "把 12 个 station repo 迁移到 monorepo" \ --verify ~/Dev/scripts/verify-monorepo.sh \ --max-iter 15 ``` **验收脚本约定** 脚本返回 JSON：`{"passing": false, "failures": ["hydro-annual build failed", "navbar missing in stack"]}` Claude 解析后优先修第一条，循环直到 `passing: true`。 **和 `/fanout-deploy` 的区别** - `/fanout-deploy`：并行执行**已知变更**，适合「我知道要改什么，让各站并行跑」 - `/autonomous-refactor`：自循环**探索性重构**，适合「我知道目标，不确定怎么到，让 Claude 自己试」 **改动范围** 新建 `~/.claude/commands/autonomous-refactor.md`（约 80 行） **当前优先级**：较低 — 需要先有成熟的验收脚本基础设施，建议等下次有大型重构任务时再立 --- ### 改进 E：夜间站群健康巡检（Scheduled） **是什么** 用 `/schedule` skill 设置一个 cron 任务，每天定时运行 `claude -p "/sites-health" --allowedTools Bash,Write`，把结果写到 `~/Dev/docs/health/site-health-YYYY-MM-DD.md`，异常时可选发通知。 **为什么加** - 27 个子域，目前每次健康检查都要手动跑 `/sites-health`，且只在「感觉有问题」时才跑 - 报告建议：「可以把每日巡检自动化，不用每次手动」 - 有了历史日志可以看趋势（哪个站最容易挂、CF 规则改完后有没有回归） **使用场景** - 每天早上 8 点自动巡检，如果有非 200/302 的站点，写入日志 - 改完 CF 配置后，次日早上自动验证是否生效 - 站群 monorepo 大改后，次日结果作为回归报告 **实现方式** 用 `/schedule` skill 创建 remote trigger： ``` /schedule create "daily site health check" --cron "0 8 * * *" \ --command "claude -p '/sites-health --json' --allowedTools Bash,Write \ --output ~/Dev/docs/health/site-health-$(date +%F).md" ``` **依赖** - `~/Dev/docs/health/` 目录需要创建 - `/sites-health` 已有 `--json` 参数，可直接用 - 需要 VPS 或本机有 cron 环境 **改动范围** - 创建 `~/Dev/docs/health/` 目录 - 用 `/schedule` skill 注册 cron trigger（非文件改动） **当前优先级**：中等 — 有用但不紧急，建议在下次 `/sites-health` 手动跑完后顺手设置 --- ## 优先级总结 | 改进 | 价值 | 成本 | 建议 | |---|---|---|---| | **A · PreToolUse Hook** | 高（防真实损失） | 低（改 settings.json 1 处） | **立刻做** | | **B · HANDOFF 放置规则** | 中（消除反复纠正） | 极低（1 行 CLAUDE.md） | **立刻做** | | **C · /fanout-deploy** | 高（27 站并行收益大） | 中（新命令 ~60 行） | **本周做** | | **D · /autonomous-refactor** | 高（大重构变过夜） | 高（需要验收脚本基建） | **下次大重构时做** | | **E · 夜间健康巡检** | 中（减少手动检查） | 低（/schedule 几分钟） | **本周做** | --- ## 执行顺序（如全部批准） ``` Step 1: 改 settings.json 加 PreToolUse hook（改进 A） Step 2: 改 CLAUDE.md 加 HANDOFF 放置规则（改进 B） Step 3: 新建 /fanout-deploy 命令文件（改进 C） Step 4: 用 /schedule 设置夜间巡检（改进 E） Step 5: 新建 /autonomous-refactor 命令文件（改进 D，可推迟） ``` **每步独立，可单独执行或跳过。**
