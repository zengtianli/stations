# Slash Command 命名空间治理（builtin 优先策略） ## Context 用户策略：**CC 内置优先级最高，user 同名版要么删除要么改名**。 研究澄清了 CC runtime 事实：**user > builtin**（runtime 里 user 版会覆盖 builtin），而非反过来。所以"让 builtin 赢"必须**主动避免命名重合**（runtime 不帮忙）。次级：**skill > command**（同名时 skill 版生效）。 调查已确认： - 冲突 3 处（`/context` + `/review` + skill 内部 `context` / `harness` 重名） - 对新增 20+ builtin（`/simplify` `/batch` `/debug` `/tasks` `/schedule` 等）user 侧**零撞名** - harness skill vs command 是**故意分工**（skill=跨项目 sync，command=单项目脚手架）— 非重复 用户已对 Q1-Q6 的推荐"都同意"，本计划落地。 目标：单批原子改名 + 全量引用同步 + 3 repo 推送。 --- ## 改名映射表 | 实体 | 原位置 | 新位置 | 理由 | |---|---|---|---| | command | `~/Dev/tools/cc-configs/commands/context.md` | `~/Dev/tools/cc-configs/commands/ctx-monitor.md` | 让 builtin `/context`（token 可视化）生效，user 版保功能改前缀 | | command | `~/Dev/tools/cc-configs/commands/review.md` | `~/Dev/tools/cc-configs/commands/review-docx.md` | 让 builtin `/review`（PR 审查）生效，对齐 `-docx` 后缀族 | | skill 目录 | `~/.claude/skills/context/` | `~/.claude/skills/zdwp-project/` | 按**归属**命名（ZDWP 水利项目），而非类型 | | skill 目录 | `~/.claude/skills/harness/` | `~/.claude/skills/harness-sync/` | 和 `/harness` command（脚手架）分工明确 | **不动的相关路径**（故意保留）： - `~/Dev/tools/cc-configs/tools/context/context.py` — 这是实现侧文件夹，非命令名；改会牵连 4 处 `tools/context/` 路径引用，收益低 - `~/Dev/tools/cc-configs/tools/context/tests/` — 同上 - cc-configs/CLAUDE.md 里 4 条 `tools/context/context.py` 路径引用 — 同上 --- ## 改动清单（按操作类型） ### A · git mv / 目录重命名（4 个） ```bash # cc-configs repo — 2 个 command 文件 cd ~/Dev/tools/cc-configs git mv commands/context.md commands/ctx-monitor.md git mv commands/review.md commands/review-docx.md # ~/.claude/skills — 2 个 skill 目录（非 git，纯文件系统 mv） mv ~/.claude/skills/context ~/.claude/skills/zdwp-project mv ~/.claude/skills/harness ~/.claude/skills/harness-sync ``` ### B · 文件内容更新（4 个文件） **B1 · `~/Dev/tools/cc-configs/commands/ctx-monitor.md`**（rename 后） -
