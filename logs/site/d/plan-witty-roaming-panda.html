# Plan · ~/Dev 死链扫描 + 基于 paths map 的修复机制 ## Context **问题**：用户反复经历"重构 → 死链"。刚立的 `paths.yaml` SSOT（1 小时前的上一任务）覆盖"注册路径"，但不检测**裸文件里指向不存在路径的引用**（"dead links"）。希望一次搞定现有死链 + 立一个机制保证"以后都不会死链"。 **Explore 发现**（82 个代表文件抽样扫，2026-04-24）： - 死链总量**仅 16 条**，集中在 3 个文件 - `tools/configs/playbooks/bid.md` — 8 条 `~/Work/bids/...`（目录未建，前瞻性引用） - `tools/cc-configs/commands/scan.md` — 3 条（`repo-dashboard/` 项目不存在 + `devtools/scripts/tools/cc_sessions.py` 应为 `lib/tools/` + `llm_client.py` 缺失） - `tools/cc-configs/skills/bid-diff-and-revise/SKILL.md` — 2 条文档与实现不一致 - 0 悬空 symlink ✓ / 0 MD `[x](path)` 死链 ✓ - 现有工具：`paths.py audit` 只查 **unregistered**（路径存在但未入 map）不查 **dead**（路径不存在）；`menus.py audit` 的 consumer-drift 模式可借鉴；无 pre-commit 路径检查 **目标**（"一步到位"的现实定义）： 1. 扩 `paths.py` 加 `scan-dead` 子命令 — **机制性**工具，以后任何时候跑一次就知道死链在哪 2. 全量扫 `~/Dev`（不只 82 个代表文件），得到完整报告 3. 按分类修复 — 3 种死链类型各有不同策略（用户拍板） 4. 扩 `paths.yaml` 加 `allow_missing:` 段（登记故意的前瞻性引用，scan-dead 排除） 5. **不做 pre-commit hook**（延续用户 "audit 手动跑" 的决定，不加机制性门卫除非用户要） --- ## 产物清单 | 文件 | 状态 | 作用 | |---|---|---| | `~/Dev/devtools/lib/tools/paths.py` | ✏️ 修改 | 加 `scan-dead` 子命令 + `allow_missing` 加载 | | `~/Dev/paths.yaml` | ✏️ 修改 | 新增 `allow_missing:` 段（本会话决定登记的前瞻引用） | | `~/Dev/tools/configs/playbooks/bid.md` | ✏️ 修复 | 8 条 `~/Work/...` 按用户决定 处理（保留 / 改 `~/Dev/work-bids` / 登记 allow_missing） | | `~/Dev/tools/cc-configs/commands/scan.md` | ✏️ 修复 | `scripts/tools/cc_sessions.py` → `lib/tools/cc_sessions.py`（验证存在后）；`llm_client.py` / `repo-dashboard` 用户决定 | | `~/Dev/tools/cc-configs/skills/bid-diff-and-revise/SKILL.md` | ✏️ 修复 | 2 条文档一致性问题 | | `~/Dev/CLAUDE.md` | ✏️ 修改 | 更新「路径注册表 SSOT」节，加 `scan-dead` 用法 | | `~/Dev/HANDOFF.md` | ✏️ 修改 | 记本轮收获：死链清零 + scan-dead 工具立 | --- ## paths.py `scan-dead` 设计 ### 子命令 ```bash python3 paths.py scan-dead # 扫全量，打印报告，exit 0（不阻塞） python3 paths.py scan-dead --strict # 有死链 exit 1（以后 hook 可用） python3 paths.py scan-dead --paths <file> # 扫指定文件 python3 paths.py scan-dead --verbose # 每条死链附 1 行上下文 ``` ### 扫描范围 **包含**（递归找 md / py / sh / yaml / json / ts / tsx）： - `~/Dev/**` - `~/.claude/CLAUDE.md` **排除**： - `_archive/`（已归档） - `migrated/`（fallback 副本） - `node_modules/` / `.git/` / `.next/` / `dist/` / `build/` - `_files.md`（auggie-map 生成物，含大量二进制列表） - `**/*.min.js` / `**/*.min.css` - `~/.personal_env`（明文不读） - 任何 `.gitignore`'d 路径（走 `git check-ignore` 判断；非 git 目录则按 hardcoded exclude list） ### 检测模式 1. **绝对路径模式**：`/Users/tianli/\S+` 或 `~/(Dev|\.claude|\.personal_env|Work)/\S+` - 去除 trailing 标点 `.,);:'"`*_>` - 末尾 `/` 去掉 - 长度 < 6 的不算 - **检查**：expanduser 后 `Path.exists()` - **例外**：在 `paths.yaml` `allow_missing:` 列表里的 prefix 匹配项跳过 2. **Markdown 链接**：`\[([^\]]+)\]\(([^)]+)\)` 的第 2 组 - 只看 path 不含 `://`（跳过 URL） - 相对路径按**文件所在目录**为 base 展开 - `~/` 或 `/` 开头按绝对路径处理 - **检查**：展开后存在？ 3. **Shell source / Python import 的硬编码路径**：`source\s+["']?(/Users/|~/)` 和 `sys.path.insert.*["'](/Users/|~/)` - 提取路径，检查存在 ### 输出格式（仿 menus.py audit） ``` [scan-dead] ~/Dev (4312 files scanned, 1823 path refs extracted) [dead / abs-path] 16 refs in 3 files bid.md:67 ~/Work/bids/<project>/delivery/ → suggest: allow_missing (forward ref) bid.md:102 ~/Work/bids/_templates/ → suggest: allow_missing scan.md:5 ~/Dev/repo-dashboard/lib/task_analyzer.py → suggest: human review scan.md:17 ~/Dev/devtools/scripts/tools/cc_sessions.py → suggest: rewrite to lib/tools/cc_sessions.py (exists) ... [dead / md-link] 0 refs [dead / source] 0 refs Summary: 16 dead / 4312 scanned (0.37%) Hot files: bid.md (8), scan.md (3), bid-diff-and-revise/SKILL.md (2) ``` ### `allow_missing` 段 schema ```yaml # paths.yaml 新增段 allow_missing: # 前瞻性引用：路径目前不存在但故意保留（未来要建） - path: ~/Work/bids reason: bid playbook forward-ref; to be built when first bid arrives added: 2026-04-24 # 其他（按需） ``` `scan-dead` 里：`startswith(allow.path)` 就跳过 + 在报告末尾单列 "allow_missing hits: N refs suppressed" 让用户知道跳过了多少。 --- ## 执行步骤 1. **扩 paths.py**（~45 min） - 加 `scan-dead` 子命令 argparse 分支 - 实现 `walk_dev()` 递归文件生成器（含 gitignore 感知：每个子目录递归时调用 `git check-ignore` 批量判断，无 .git 目录则按 fallback exclude list） - 实现 3 类检测函数 + `allow_missing` 过滤 - 实现报告打印 + exit 1 strict 模式 - 更新 `load_paths()` 返回的 `data` 里处理 `allow_missing` 2. **加 `allow_missing:` 段到 paths.yaml**（已知需要登记的：`~/Work/bids` 前瞻引用） 3. **首跑 scan-dead** - `python3 devtools/lib/tools/paths.py scan-dead > /tmp/dead.txt` - 对比 Explore 抽样的 16 条，看全量扫是不是真有更多 4. **出报告给用户 + 拍板修复策略** - 按死链分类分组（abs-path / md-link / source） - 每类列 3 种修法：delete / rewrite / allow_missing - 用 AskUserQuestion 让用户批量选（每组一个决定） 5. **执行修复** - 自动安全修复：`scripts/tools/cc_sessions.py` → `lib/tools/cc_sessions.py` 这种路径笔误（前提：验证目标存在） - 用户决定的批量改动：按第 4 步的回答 sed / Edit - 真要删的行：Edit 掉 6. **二次 scan-dead 验证清零** - 预期 `Summary: 0 dead / N scanned` 7. **更新 CLAUDE.md**：「路径注册表 SSOT」节加 `scan-dead` 用法 8. **更新 HANDOFF.md**：记 "死链清零 + scan-dead 工具上线" 9. **commit**（两个 repo）： - `~/Dev/devtools` @ `feat/paths-ssot`: paths.py 扩 + scan-dead - `~/Dev` @ `main`: paths.yaml / CLAUDE.md / HANDOFF.md - 被修的 playbooks / commands / skills 文件根据所属 repo 分别 commit --- ## "以后都不死链"的机制说明 **本轮的机制性贡献**： - `paths.py scan-dead` 是可重复调用的工具 — 每次重构后跑一次就知道死链在哪 - `allow_missing:` 段让故意的前瞻引用可登记可审计 - paths.yaml 本身鼓励新文档 / 脚本 reference map 而非裸路径，从源头减少死链 **不做**（避免过度工程）： - ❌ 不装 pre-commit hook（延续用户"audit 手动跑"决定） - ❌ 不全局 sed 旧裸路径（保守策略延续，auto-fix 只限"笔误"级无歧义改动） - ❌ 不做 fuzzy rename 建议（scan-dead 只报告 + 1 级人工判断提示，不猜测） **下一步升级路径**（本轮不做，记入 HANDOFF）： - scan-dead `--fix-safe` 标志：只应用"明确的笔误修正"（目标存在 + 差异 ≤ 3 字符 + 唯一匹配） - 接入 devtools 的 pre-commit（可选，默认不开） - 接入 `/health` skill 作为第 N 项检查 --- ## 验证（end-to-end） ```bash cd ~/Dev # 1. 工具功能 python3 devtools/lib/tools/paths.py scan-dead | head -50 python3 devtools/lib/tools/paths.py scan-dead --strict && echo "CLEAN" # 2. 单文件扫 python3 devtools/lib/tools/paths.py scan-dead --paths tools/configs/playbooks/bid.md # 3. 修复后复跑应 0 条死链（或只有 allow_missing 登记的） python3 devtools/lib/tools/paths.py scan-dead # 预期：Summary: 0 dead / N scanned, M refs suppressed by allow_missing # 4. audit 也复跑确认 paths.yaml 加 allow_missing 后没坏 python3 devtools/lib/tools/paths.py audit ``` --- ## 明确不做（范围界限） - ❌ 不扫 `_archive/` / `migrated/` / `node_modules/` / `.git/` / 其他 gitignored - ❌ 不做 fuzzy 自动改名（只做明确笔误级改动，其余报告让用户决定） - ❌ 不接 pre-commit hook（除非用户本轮主动要求） - ❌ 不删除无法确定归宿的死链行（留报告给用户人工判断） - ❌ 不碰 `_files.md` / `.min.js` / `.min.css`（大文件 / 生成物） --- ## 回滚 所有变动都在 git 追踪下（devtools repo + ~/Dev meta repo + 具体被修的子 repo）： ```bash cd ~/Dev/devtools && git checkout main -- lib/tools/paths.py cd ~/Dev && git checkout HEAD -- paths.yaml CLAUDE.md HANDOFF.md # 被修的子 repo 各自 git checkout ```
